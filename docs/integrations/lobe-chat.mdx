# Lobe Chat

Lobe Chat 和井陉的组合很适合两类场景：

- 你想要一个更完整的聊天前端，但底层模型统一走井陉
- 你想在自部署环境里，把 OpenAI 兼容网关作为统一接入点

这页需要分成两种情况看：**官方托管版** 和 **自部署版**。

## 先说结论

- 想完全掌控 Base URL 和 API Key：优先用自部署版
- 想快速体验：先确认你当前使用的 Lobe Chat 形态是否允许配置 OpenAI provider
- 想减少后续混乱：给 Lobe Chat 单独发令牌
- 想少踩坑：先只验证 OpenAI provider 路径，不要一开始混太多 provider

## 接入前准备

无论你用哪一种方式，先准备好：

- Base URL：`https://api.jingxing.uk/v1`
- API Key：一个专门给 Lobe Chat 使用的井陉令牌
- 模型名：当前分组下已经验证过的模型

## 情况一：官方托管版怎么理解

Lobe Chat 官方托管版本身支持在设置中配置部分模型服务商。  
如果你当前使用的是面向最终用户的托管版界面，那么最重要的问题不是“能不能输入 token”，而是**它是否允许你真正覆盖 OpenAI provider 的 API key 与 base URL**。

因此更稳的建议是：

- 如果只是个人快速体验，先看当前版本设置页是否允许填 OpenAI provider
- 如果你明确要把底层流量统一走井陉，优先考虑自部署版

原因很简单：井陉的价值之一就是“统一网关和统一治理”，而这通常更适合放在你能控制环境变量或服务端配置的地方。

## 情况二：自部署版怎么接

这是更推荐的路径。  
根据 LobeHub 官方自部署环境变量文档，Lobe Chat 支持通过 OpenAI 相关环境变量来接入兼容接口，其中核心就是：

- `OPENAI_API_KEY`
- `OPENAI_PROXY_URL`

对井陉来说，这两项通常就够你先跑通核心聊天链路。

## 推荐接入方式

### 1. 先在井陉创建一个 Lobe Chat 专用令牌

推荐配置：

- 名称：`lobe-web`
- 分组：适合聊天前端场景的分组
- 模型限制：先只开放要给 Lobe Chat 用的少数模型
- 额度：保证足够完成首次验证

### 2. 自部署版设置环境变量

如果你是 Docker、Docker Compose 或其他自部署方式，优先配置：

```bash
OPENAI_API_KEY=<YOUR_JINGXING_TOKEN>
OPENAI_PROXY_URL=https://api.jingxing.uk/v1
```

如果你的部署还需要显式控制模型列表，也可以继续补模型相关变量，但第一轮不建议把事情做得太复杂。

### 3. 进入 Lobe Chat 后只验证一个模型

第一轮不要一口气塞很多模型。  
更稳的做法是：

- 先确认 OpenAI provider 已启用
- 先确认模型列表能看到目标模型
- 只选一个模型发起第一条对话

## 推荐模型选择思路

Lobe Chat 更像通用聊天前端，所以模型选择优先级通常是：

1. 对话稳定
2. 响应速度适中
3. 长上下文表现可接受
4. 成本在你的预期内

如果只是验证接入，不要先拿高成本模型做第一步。

## 为什么推荐单独给 Lobe Chat 发令牌

因为它是一个长期在线的聊天前端，不像单次命令行请求那么简单。  
它可能持续保存会话、频繁切换模型、被多人使用，或者接到不同浏览器会话。

如果和其他客户端共用令牌，后面很难判断：

- 到底是谁在消费额度
- 到底哪个客户端在报错
- 到底哪个模型在被频繁调用

## 最短验证路径

1. 先用命令行跑通 `GET /models`
2. 配置 `OPENAI_API_KEY` 与 `OPENAI_PROXY_URL`
3. 启动或重启 Lobe Chat
4. 在界面里只选一个模型
5. 发起一条最简单的对话

例如：

- “用一句话介绍井陉”
- “帮我整理这段文本重点”

## 常见问题

### 模型列表不对

优先检查：

- 当前是不是还在用旧环境变量
- 当前 token 是否限制了模型
- 当前分组是否开放该模型

### 页面能打开，但聊天失败

优先检查：

- `OPENAI_PROXY_URL` 是否写成了 `https://api.jingxing.uk/v1`
- `OPENAI_API_KEY` 是否是当前有效令牌
- 当前模型是否已验证可用

### 自部署版更新后行为变了

这通常不是井陉专属问题，而是前端版本、provider 设置或环境变量处理方式变化。  
出现这种情况时，先用最小变量组合重新验证，不要一次性带太多历史配置。

## 推荐实践

- 自部署版优先
- 先只用 OpenAI provider 路径
- 先少量模型、少量变量
- 给 Lobe Chat 单独发令牌
- 出问题先回命令行验证 API
