# Lobe Chat

Lobe Chat 适合构建完整的 Web 聊天前端，并通过 OpenAI 兼容接口统一接入井陉。对于井陉而言，Lobe Chat 的推荐方式为自部署方案；若使用官方托管版本，则应先确认当前版本是否允许完整覆盖 OpenAI provider 的 API Key 与 Base URL。

## 适用场景

- 需要完整的聊天前端，并统一接入井陉
- 需要在自部署环境中管理模型、令牌和访问路径
- 需要为后续图像、音频或 RAG 能力扩展预留空间

## 前置条件

请先准备以下信息：

- Base URL：`https://api.jingxing.uk/v1`
- API Key：专门用于 Lobe Chat 的井陉令牌
- 模型名称：当前分组下已经验证可用的模型

## 部署形态说明

### 官方托管版

若使用 Lobe Chat 官方托管版本，应优先确认当前设置页面是否允许完整配置 OpenAI provider 的 API Key 与 Base URL。若当前版本仅允许有限配置，则不适合作为统一接入井陉的长期方案。

### 自部署版

自部署版是更推荐的接入方式。根据 LobeHub 官方文档，Lobe Chat 可通过 OpenAI 相关环境变量接入兼容接口，其中最关键的变量为：

- `OPENAI_API_KEY`
- `OPENAI_PROXY_URL`

## 配置步骤

### 1. 在井陉侧创建 Lobe Chat 专用令牌

建议配置如下：

- 名称：如 `lobe-web`
- 分组：适合 Web 聊天前端的分组
- 模型限制：仅开放计划在 Lobe Chat 中使用的模型
- 额度：覆盖验证与日常使用需求

### 2. 在自部署环境中设置变量

```bash
OPENAI_API_KEY=<YOUR_JINGXING_TOKEN>
OPENAI_PROXY_URL=https://api.jingxing.uk/v1
```

首次接入建议仅保留以上最小变量组合，避免同时引入过多历史配置或其他 provider 设置。

### 3. 首次仅验证一个模型

启动或重启服务后，建议仅选择一个模型进行验证，确认模型列表、令牌权限与主链路均正常后，再扩展更多模型或能力。

## 模型选择建议

Lobe Chat 作为通用聊天前端，建议优先考虑：

1. 对话稳定性
2. 响应速度
3. 长上下文表现
4. 成本可控性

首次接入不建议直接选择高成本模型。

## 验证方法

建议按以下顺序完成验证：

1. 在命令行中使用当前令牌执行 `GET /v1/models`
2. 配置 `OPENAI_API_KEY` 与 `OPENAI_PROXY_URL`
3. 启动或重启 Lobe Chat
4. 在界面中仅选择一个模型
5. 发起一次简单对话请求

## 常见问题

### 模型列表异常

建议优先检查：

- 当前环境变量是否已生效
- 当前令牌是否限制了模型
- 当前分组是否开放目标模型

### 页面可访问，但聊天失败

建议排查：

- `OPENAI_PROXY_URL` 是否准确填写为 `https://api.jingxing.uk/v1`
- `OPENAI_API_KEY` 是否为当前有效令牌
- 当前模型是否已经完成独立验证

### 自部署升级后行为变化

此类问题通常与前端版本、provider 配置方式或环境变量处理变化有关。建议先使用最小变量组合重新验证，再逐步恢复其他配置。

## 实践建议

- 优先使用自部署版
- 首先完成 OpenAI provider 主链路验证
- 为 Lobe Chat 单独签发令牌
- 首次仅配置少量模型
- 出现问题时，优先以命令行请求排除 API 侧问题
