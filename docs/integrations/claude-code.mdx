# Claude Code

Claude Code 是 Anthropic 提供的终端编码工具。对于井陉而言，推荐的接入方式并非让每位开发者直接维护多家上游凭据，而是在组织内部通过企业代理或 LLM Gateway 统一接入，再由网关层路由至井陉。

## 适用场景

- 团队已经具备 Claude Code 的企业代理或网关方案
- 希望统一管理鉴权、审计、模型路由与成本
- 需要在 Claude Code 之外，同时管理其他模型渠道

## 基本原则

Claude Code 的官方支持路径通常包括：

- Anthropic 官方 API
- Amazon Bedrock
- Google Vertex AI
- 企业代理或 LLM Gateway

井陉在此体系中更适合作为“统一模型网关”存在，而不是每个终端直接维护多个上游供应商配置。

## 推荐架构

```text
Claude Code
  -> 企业代理 / LLM gateway
    -> Jingxing
      -> Claude 渠道与其他模型渠道
```

该方式的主要收益包括：

- 统一鉴权与权限管理
- 统一调用日志与用量审计
- 统一模型暴露范围与分组策略

## 前置条件

开始配置前，建议确认以下条件已经满足：

1. 井陉侧已启用至少一个 Claude 模型渠道
2. 当前分组允许对应模型对外可见
3. 已为团队或环境准备专用令牌
4. Claude Code 当前部署方式支持企业代理或 LLM Gateway

## 井陉侧准备步骤

### 1. 确认 Claude 能力可用

在控制台中检查以下信息：

- Claude 渠道处于启用状态
- 分组已开放目标模型
- 令牌具备对应调用权限

### 2. 创建独立令牌

建议为 Claude Code 所在环境单独创建令牌，避免与个人测试、桌面客户端或生产 API 服务混用。

### 3. 在组织网关层对接井陉

若团队已经存在内部 LLM Gateway，建议在该层统一接入井陉，并由该层转发 Claude Code 请求。该方式优于让每位开发者直接管理不同上游配置。

## Claude Code 侧配置建议

建议采用以下做法：

- 使用 Claude Code 的最新稳定版本
- 按 Anthropic 官方文档启用企业代理或 LLM Gateway
- 通过团队统一出口访问井陉

不建议采用以下做法：

- 每台机器单独维护上游厂商私钥
- 将生产令牌分散在个人终端环境中
- 将临时验证方案直接用于团队级环境

## 验证方法

接入完成后，可使用简单请求进行验证，例如：

```bash
claude -p "用三句话总结当前仓库的用途。"
```

验证时建议同时确认：

- Claude Code 是否可以正常发起请求
- 井陉侧是否能够记录到调用日志
- 分组、额度和模型权限是否按预期生效

## 常见问题

### 请求未经过井陉

此类问题通常说明 Claude Code 仍在使用默认官方路径。应优先检查企业代理或网关配置是否已生效。

### 令牌有效，但调用仍失败

建议依次排查：

- Claude Code 当前部署方式是否支持该接入方案
- 企业代理是否正确转发到井陉
- 网络、证书或组织策略是否阻断请求

### 团队内只有部分成员可用

此类问题通常与环境变量、代理设置、证书或企业网络策略差异有关。建议先统一入口，再排查个别终端差异。

## 实践建议

- 采用企业代理或网关层统一接入
- 为团队环境单独签发令牌
- 在井陉侧单独管理 Claude 模型分组
- 以组织级治理方式管理调用策略和审计边界
