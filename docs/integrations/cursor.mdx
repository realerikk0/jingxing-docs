# Cursor

Cursor 是一款面向开发者的 AI 代码编辑器，基于 VS Code 体验构建，支持代码问答、文件改写、Inline Edit、Chat、Agent 等开发场景。接入井陉后，可以在 Cursor 中通过 OpenAI 兼容接口调用井陉提供的模型，并使用井陉令牌统一管理鉴权、额度和调用记录。

## 下载 Cursor

如本机尚未安装 Cursor，请先从官方渠道下载：

- [Cursor 官方下载](https://cursor.com/en-US/download)
- [Cursor 安装说明](https://docs.cursor.com/en/get-started/installation)

安装完成后，打开 Cursor，并确认可以进入 `Cursor Settings`。

## 适用场景

- 希望在 IDE 中快速接入井陉模型。
- 希望使用 Cursor 进行代码解释、代码生成、文件改写和项目问答。
- 希望将 Cursor 的模型调用与其他客户端分开管理。
- 希望通过单独令牌控制 Cursor 的额度、权限和调用记录。

## 前置条件

开始前，请先准备以下信息：

| 项目 | 说明 |
| --- | --- |
| Cursor | 已安装并可以正常打开 |
| API Base URL | `https://api.jingxing.uk/v1` |
| API Key | 在“令牌管理”中创建的 `sk-` 开头令牌 |
| 模型名称 | 当前令牌可用的模型名称，以控制台显示为准 |

建议为 Cursor 单独创建令牌，不要复用生产服务、团队共享或其他客户端正在使用的高权限令牌。

## 接入流程

```text
创建 Cursor 专用令牌
  -> 确认可用模型
  -> 打开 Cursor Settings
  -> 填写 API Key 和 Base URL
  -> 添加或选择模型
  -> 发送一次测试请求
```

## 如何接入

### 1. 创建 Cursor 专用令牌

进入井陉控制台：

```txt
令牌管理 -> 新建令牌
```

建议命名为 `cursor-dev`、`cursor-mac`、`cursor-work` 等便于识别的名称。后续如果需要排查 Cursor 消耗、停用某台设备或调整权限，可以直接定位到对应令牌。

### 2. 确认模型可用

在控制台中查看当前令牌可用的模型。首次接入建议只选择一个模型完成验证，确认 Cursor 和井陉之间的链路正常后，再增加其他模型。

模型名称需要与控制台显示保持一致。例如：

```txt
gpt-4.1
```

如使用其他模型，请以当前控制台中可见、可调用的名称为准。

### 3. 打开 Cursor 模型设置

在 Cursor 中进入：

```txt
Cursor Settings -> Models
```

找到 OpenAI 或 OpenAI API Key 相关配置区域。不同 Cursor 版本的界面名称可能略有差异，但通常需要完成以下三项：

- 启用 OpenAI API Key。
- 填写井陉 API Key。
- 启用或填写 Override OpenAI Base URL。

### 4. 填写井陉配置

在 Cursor 的 OpenAI 兼容配置中填写：

| 字段 | 填写内容 |
| --- | --- |
| OpenAI API Key | 井陉令牌，例如 `sk-...` |
| Override OpenAI Base URL | `https://api.jingxing.uk/v1` |
| Model | 当前令牌可用的模型名称 |

填写完成后，点击 Cursor 界面中的 `Verify`、`Save` 或对应确认按钮。若当前版本提供连接测试，建议先完成验证再进入正式会话。

### 5. 添加或选择模型

如果模型列表中已经出现目标模型，直接选择即可。

如果列表中没有目标模型，可以尝试使用 `Add Custom Model` 添加模型名称。添加时请使用井陉控制台中显示的完整模型名，不要自行改写大小写、前缀或后缀。

首次验证建议选择一个普通聊天或编码模型，不建议同时添加大量模型。这样更容易判断问题来自模型名称、令牌权限，还是 Cursor 配置本身。

### 6. 发送测试请求

打开一个项目目录，在 Cursor Chat 中发送一个简单请求：

```txt
请总结当前文件的作用，并指出可以改进的地方。
```

如果能够正常返回，并且井陉控制台中出现对应调用记录，说明接入已经完成。

## 验证方法

推荐按以下顺序验证：

1. 在井陉控制台确认令牌处于启用状态。
2. 在 Cursor 中填写 API Key 和 Base URL。
3. 点击 `Verify` 或保存配置。
4. 添加或选择一个模型。
5. 发送一条简单 Chat 请求。
6. 回到井陉控制台查看调用记录和额度消耗。

## 注意事项

- Cursor 自定义 API Key 主要适用于标准聊天模型。
- Tab Completion 等依赖 Cursor 专用模型的能力，可能仍会使用 Cursor 内置模型。
- Override OpenAI Base URL 可能影响 Cursor 对 OpenAI 模型的整体路由。
- 如果需要同时使用 Cursor 原生模型和井陉模型，建议保留一个清晰的切换流程。

## 常见问题

### Base URL 应该填写什么

填写：

```txt
https://api.jingxing.uk/v1
```

Cursor 通过 OpenAI 兼容接口接入井陉，因此这里需要包含 `/v1`。

### API Key 验证失败

请依次检查：

- API Key 是否完整复制，且以 `sk-` 开头。
- Base URL 是否填写为 `https://api.jingxing.uk/v1`。
- 当前令牌是否已启用。
- 当前令牌是否还有可用额度。
- 当前网络是否可以访问井陉 API。

### 模型列表中没有目标模型

可以尝试使用 `Add Custom Model` 手动添加模型名称。模型名称必须与井陉控制台显示一致。

如果添加后仍不可用，建议先换用一个已确认可调用的模型完成验证，再排查目标模型本身的权限和兼容性。

### Chat 可以用，但 Tab Completion 没有变化

这是 Cursor 的正常限制之一。自定义 API Key 主要用于标准聊天模型，Tab Completion 等专用能力可能继续使用 Cursor 内置模型，不一定会通过井陉令牌调用。

### 开启 Override OpenAI Base URL 后，其他模型异常

该设置可能会影响 Cursor 对 OpenAI 相关模型的整体路由。如果需要频繁切换 Cursor 原生模型和井陉模型，建议在切换前确认当前 Base URL、API Key 和模型选择是否一致。

### 如何确认请求是否经过井陉

最直接的方式是在井陉控制台查看调用记录。若 Cursor 返回成功，但控制台没有记录，通常说明当前请求没有使用井陉的 API endpoint，或 Cursor 仍在使用原有模型配置。
