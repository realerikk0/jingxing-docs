# Cursor

Cursor 支持通过自定义 OpenAI 兼容配置接入第三方网关。对井陉而言，Cursor 是优先级较高的编码客户端之一，适合用于交互式编码、代码解释、文件改写和日常开发辅助。

## 适用场景

- 需要以最少配置完成 IDE 接入
- 需要在编码场景中统一使用井陉提供的模型能力
- 需要将编码流量与其他聊天流量分开管理

## 前置条件

开始配置前，请确认以下信息已经准备完成：

- Base URL：`https://api.jingxing.uk/v1`
- API Key：专门用于 Cursor 的井陉令牌
- 模型名称：当前分组下已经验证可用的模型

建议避免直接复用以下令牌：

- 命令行测试令牌
- 团队共享的生产令牌
- 未做模型限制的大权限令牌

## 配置步骤

### 1. 在井陉侧创建 Cursor 专用令牌

建议采用以下配置原则：

- 名称：如 `cursor-dev`、`cursor-team`
- 分组：选择适合编码和交互式对话的分组
- 模型限制：仅开放计划在 Cursor 中使用的模型
- 额度：满足验证与日常使用需求

### 2. 在 Cursor 中填写自定义 OpenAI 配置

根据 Cursor 的 BYOK 配置路径，填写以下信息：

- API Key：井陉令牌
- Base URL：`https://api.jingxing.uk/v1`

如当前版本提供连接测试或验证按钮，建议在保存配置后先执行一次验证。

### 3. 首次仅验证一个模型

首次接入建议仅使用一个已经验证可用的模型，以便快速判断接入链路是否正常。完成首次验证后，再逐步补充其他模型。

## 模型选择建议

Cursor 的典型使用场景是交互式编码，因此建议优先关注以下因素：

1. 响应稳定性
2. 长上下文能力
3. 工具调用与复杂输出能力
4. 成本可控性

首次接入不建议直接选择成本最高的模型，应优先选择已经在命令行环境中完成验证的常规模型。

## 验证方法

推荐按以下顺序完成验证：

1. 在命令行中使用当前令牌执行 `GET /v1/models`
2. 在 Cursor 中填入 API Key 与 Base URL
3. 选择一个已验证可用的模型
4. 执行一次简单请求，例如解释当前文件、补充注释或总结目录结构

如上述步骤可以完成，则说明核心接入链路已经建立。

## 已知限制

截至 2026-04-15，Cursor 对自定义 OpenAI Base URL 的支持已可用于常规接入，但在部分版本中，覆盖 OpenAI Base URL 与 API Key 后，可能会影响 Cursor 原生模型或其他提供商模型的切换体验。

这意味着：

- 若 Cursor 主要用于接入井陉，则通常不会构成障碍
- 若同一实例需要频繁切换井陉模型与 Cursor 原生模型，则应预留额外测试时间

## 常见问题

### 401 未通过认证

建议依次检查以下内容：

- API Key 是否完整复制
- Base URL 是否准确写为 `https://api.jingxing.uk/v1`
- 当前令牌是否过期、停用或失效

### 模型可见但调用失败

建议优先排查：

- 当前令牌是否对模型做了限制
- 当前分组是否开放该模型
- Cursor 是否仍缓存旧配置

### 开启自定义 Base URL 后其他模型不可用

此类问题通常与 Cursor 当前版本的 BYOK 行为有关，不一定是井陉侧问题。若需要混合使用多类模型，建议先明确当前工作流的主要模型来源，并减少频繁切换。

### 响应缓慢或频繁超时

建议优先检查：

- 当前分组是否偏向低成本而非低延迟
- 当前模型是否适合交互式编码
- 当前会话上下文是否过长

## 实践建议

- 为 Cursor 单独签发令牌
- 为 Cursor 选择独立分组
- 首次仅接入少量模型
- 先完成命令行验证，再完成客户端验证
- 出现问题时，优先使用最小请求排除网关侧问题
