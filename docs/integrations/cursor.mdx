# Cursor

Cursor 是井陉最值得优先支持的开发工具之一，因为它本身就有比较成熟的 OpenAI 兼容接入路径。  
如果你的目标是“用最少配置把 IDE 接到井陉”，Cursor 通常会比 Claude Code 更直接。

## 先说结论

- 想要最快跑通：优先用 Cursor 接井陉
- 想要最稳定：给 Cursor 单独发一个令牌
- 想要少踩坑：先只验证一个明确可用的模型
- 想混用 Cursor 原生模型和自定义 OpenAI Base URL：要提前接受当前版本仍可能存在切换限制

## 接入前准备

开始前，先准备好这三项：

- Base URL：`https://api.jingxing.uk/v1`
- API Key：一个专门给 Cursor 使用的令牌
- 模型名：当前分组下实际可见、并且你已经确认可用的模型

建议不要直接复用：

- 本地 `curl` 测试令牌
- 团队共享生产令牌
- 没有模型限制的大权限令牌

## 为什么建议给 Cursor 单独发令牌

因为 Cursor 不是单纯的“发一条请求就结束”的客户端。  
它可能会：

- 先拉模型列表
- 频繁切模型
- 在不同功能里复用同一套配置
- 在网络不稳定时触发重试

如果你把 Cursor 和其他客户端共用一个令牌，后面在排查额度、模型选择、超时和异常流量时会非常难拆。

## 推荐接入方式

### 1. 先在井陉侧准备一个 Cursor 专用令牌

推荐配置：

- 名称：`cursor-dev` 或 `cursor-team`
- 分组：选择适合交互式编码的分组
- 额度：先给足够跑通的额度
- 模型限制：只开放你打算在 Cursor 中使用的几个模型

### 2. 在 Cursor 里填写自定义 OpenAI 配置

按 Cursor 官方的 BYOK 文档思路，找到与 OpenAI API Key / Base URL 相关的设置项，然后填入：

- API Key：你的井陉令牌
- Base URL：`https://api.jingxing.uk/v1`

如果你的版本支持验证按钮，可以先做一次验证；如果没有，也可以直接在聊天或模型列表里验证。

### 3. 只先验证一个模型

第一轮不要一下子加很多模型，建议先只用一个你已经确认可用的模型，例如：

- 一个通用聊天模型
- 一个你已经在 `GET /models` 里见过的模型

等这一步稳定后，再考虑补更多模型。

## 推荐模型选择思路

Cursor 主要是交互式编码场景，所以优先级通常是：

1. 响应速度稳定
2. 长上下文下仍然可靠
3. 对工具调用和长输出更友好
4. 成本在可接受范围内

如果你只是先验证连通性，不要一上来就选最贵的模型。

## 当前要注意的一个限制

截至 2026-04-15，Cursor 官方有 BYOK 接入路径，但社区里仍有关于自定义 OpenAI Base URL 的已知限制：当你覆盖 OpenAI Base URL/Key 后，这套配置可能会影响其他模型提供商或 Cursor 原生模型的切换行为。

这意味着：

- 如果你主要就是把 Cursor 当作“接井陉的客户端”，通常没有问题
- 如果你想在同一个 Cursor 实例里频繁来回切“井陉自定义模型”和“Cursor 原生模型”，当前版本可能不够丝滑

更务实的做法是：

- 需要用井陉时，明确使用自定义 OpenAI 配置
- 需要切回其他原生模型时，按当前版本实际行为决定是否关闭 override 设置

## 最短验证路径

1. 先确认 `GET /models` 在命令行可用
2. 再在 Cursor 中填入 API Key 和 Base URL
3. 只选一个模型做第一次对话
4. 让它执行一个最简单的编码任务

例如：

- “解释当前文件做什么”
- “给这个函数补一个更短的注释”
- “列出这个项目的主要目录用途”

如果这一步成功，说明最关键的接入已经打通。

## 常见问题

### 填了 Key 但一直 401

优先检查：

- API Key 是否复制完整
- Base URL 是否写成了 `https://api.jingxing.uk/v1`
- 当前令牌是否已经过期或停用

### 模型能看到，但一调用就失败

优先检查：

- 当前令牌是否限制了模型
- 当前分组是否真的支持该模型
- Cursor 是否还缓存着旧模型配置

### 开了自定义 Base URL 后，别的模型变得不可用

这不是井陉专属问题，而是 Cursor 当前 BYOK / Base URL override 路径的已知限制之一。  
如果你必须混用不同来源的模型，建议先明确“当前这一轮主要用哪一类模型”，不要频繁来回切。

### 一直超时或体验很卡

优先检查：

- 当前分组是否更适合低成本而不是低延迟
- 当前模型是否不适合交互式编码
- 当前对话上下文是否已经过长

## 推荐实践

- 给 Cursor 单独发令牌
- 给 Cursor 单独放一个适合交互式编码的分组
- 模型列表先少后多
- 先验证一个模型，再慢慢扩
- 遇到问题先用命令行最小请求排除平台侧问题
