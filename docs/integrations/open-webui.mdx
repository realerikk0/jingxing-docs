# Open WebUI

Open WebUI 原生支持 OpenAI-compatible API，适合作为井陉的自部署 Web 前端入口。其优势在于可以将聊天、RAG、图像和音频等能力分层管理，因此适合需要逐步扩展功能的团队环境。

## 适用场景

- 需要统一的自部署 Web 前端
- 需要通过 OpenAI 兼容方式接入井陉
- 需要后续扩展 RAG、图像或音频能力

## 前置条件

请先准备以下信息：

- Base URL：`https://api.jingxing.uk/v1`
- API Key：专门用于 Open WebUI 的井陉令牌
- 模型名称：至少一个已经验证可用的模型

## 配置原则

Open WebUI 的推荐接入顺序为：

1. 先建立聊天主链路
2. 再扩展 RAG / Embedding
3. 最后扩展图像与音频能力

该顺序有助于将主链路验证与扩展能力验证分开处理，降低排查复杂度。

## 配置步骤

### 1. 在井陉侧创建 Open WebUI 专用令牌

建议配置如下：

- 名称：如 `openwebui-web`
- 分组：适合通用聊天或生产前端的分组
- 模型限制：首次仅开放聊天所需模型
- 额度：覆盖验证与日常使用需求

### 2. 通过环境变量配置聊天主链路

```bash
OPENAI_API_BASE_URL=https://api.jingxing.uk/v1
OPENAI_API_KEY=<YOUR_JINGXING_TOKEN>
```

如通过管理面板配置，也建议首次仅填写 Base URL 与 API Key，不建议在首轮验证时同时配置图像、音频或 RAG 相关参数。

### 3. 仅验证一个模型

启动或重启服务后，先确认模型列表中出现目标模型，并仅使用一个模型发起首次对话请求。

## 扩展能力说明

### RAG / Embedding

在聊天主链路稳定后，可单独扩展向量化与检索链路。扩展前建议先确认井陉当前分组与模型中已开放相应 Embedding 能力。

### 图像能力

若需要在 Open WebUI 中启用图像生成，应先确认当前分组中已开放图像模型，再配置相应 endpoint 与 key。

### 音频能力

音频能力同样建议在聊天主链路稳定后再单独启用，并先完成独立的 TTS / STT 接口验证。

## 验证方法

建议按以下顺序完成验证：

1. 在命令行中使用当前令牌执行 `GET /v1/models`
2. 配置 `OPENAI_API_BASE_URL` 与 `OPENAI_API_KEY`
3. 重启 Open WebUI
4. 在界面中选择一个模型
5. 发起一次简单对话请求

## 常见问题

### 模型未显示

建议优先检查：

- 当前令牌是否具备模型权限
- Base URL 是否准确写为 `https://api.jingxing.uk/v1`
- Open WebUI 是否仍在使用旧配置

### 聊天可用，但图像或音频不可用

此类问题通常说明图像或音频能力尚未开放，或尚未完成对应能力的独立配置，并不一定表示聊天接入存在问题。

### 认证失败或权限不足

建议依次检查：

- API Key 是否有效
- 令牌是否过期或停用
- 分组是否开放目标模型
- 来源限制是否误伤当前环境

## 实践建议

- 先完成聊天主链路验证
- 为 Open WebUI 单独签发令牌
- 首次仅配置少量模型
- 扩展能力采用分阶段启用方式
- 出现问题时，优先排除聊天主链路，再排查扩展功能
