# Cherry Studio

Cherry Studio 支持通过自定义服务商接入 OpenAI 兼容接口，适合将井陉作为桌面客户端的统一模型入口。对于井陉而言，推荐采用“自定义服务商 + OpenAI 类型”的方式配置，而非混入已有官方服务商配置。

## 适用场景

- 需要快速完成桌面客户端接入
- 需要将井陉与官方 OpenAI 等服务商清晰区分
- 需要在同一桌面客户端中管理多个用途不同的服务实例

## 前置条件

开始配置前，请准备以下信息：

- Base URL：`https://api.jingxing.uk/v1`
- API Key：专门用于 Cherry Studio 的井陉令牌
- 模型名称：当前分组下已经验证可用的模型

建议避免直接复用以下令牌：

- Cursor 使用中的令牌
- 团队生产服务令牌
- 未限制模型范围的大权限令牌

## 配置原则

推荐使用自定义服务商的原因如下：

- 服务商名称可以直接标识为 `Jingxing`
- 可以与已有官方服务商配置分离
- 后续切换令牌、模型和分组时更易于维护

## 官方界面示意

以下界面示意来自 Cherry Studio 官方文档，用于说明“自定义服务商”配置入口与服务商类型选择方式。实际界面可能随版本变化，但关键字段保持一致。

![Cherry Studio 自定义服务商示意](/img/integrations/cherry-custom-provider-doc.png)

图中应重点关注以下信息：

- 配置入口位于模型服务商设置页
- 新增服务商时应选择 `OpenAI` 类型
- 令牌与 API 地址需分别填写，不建议混用已有官方服务商配置
- 完成服务商新增后，再手动补充需要的模型

## 配置步骤

### 1. 在井陉创建 Cherry Studio 专用令牌

建议配置如下：

- 名称：如 `cherry-desktop`、`cherry-team`
- 分组：适合桌面对话的聊天分组
- 模型限制：仅开放计划在 Cherry Studio 中使用的模型
- 额度：覆盖验证与日常使用需求

### 2. 在 Cherry Studio 中新增自定义服务商

按照 Cherry Studio 官方文档的自定义服务商流程：

1. 新增服务商
2. 服务商类型选择 `OpenAI`
3. 服务商名称填写 `Jingxing` 或组织内部标准命名

### 3. 填写接口地址与令牌

在服务商配置中填写：

- API Key：井陉令牌
- API 地址：`https://api.jingxing.uk/v1`

如当前版本提供检查或测试按钮，建议保存后先执行一次连通性验证。

### 4. 手动添加模型

首次接入建议只添加少量已经验证可用的模型，例如一个常规聊天模型和一个备用模型。待主链路稳定后，再逐步扩充模型列表。

## 多用途配置建议

若 Cherry Studio 同时用于开发、生产观察或高成本模型测试，建议直接创建多个服务商实例，而不是频繁修改同一配置。例如：

- `Jingxing Dev`
- `Jingxing Prod`
- `Jingxing HighCost`

该方式更有利于长期维护，也便于分离不同用途的令牌与分组。

## 验证方法

建议按以下顺序完成验证：

1. 在命令行中使用当前令牌执行 `GET /v1/models`
2. 在 Cherry Studio 中新增 `Jingxing` 服务商
3. 填写 API Key 与 Base URL
4. 手动添加一个已验证模型
5. 发起一次简单对话请求

## 常见问题

### 服务商可新增，但模型不可用

建议优先检查：

- 模型是否已手动添加
- 模型名称是否与井陉当前可见模型一致
- 当前分组是否开放该模型

### 认证失败

建议依次检查：

- API Key 是否完整复制
- API 地址是否准确填写为 `https://api.jingxing.uk/v1`
- 当前令牌是否过期、停用或无效

### 响应超时或失败率偏高

建议检查：

- 当前分组是否偏向低成本而非稳定性
- 当前模型是否适合长上下文场景
- Cherry Studio 是否存在频繁重试

## 实践建议

- 为 Cherry Studio 单独签发令牌
- 使用自定义服务商而非混用官方服务商配置
- 首次仅配置少量模型
- 不同用途使用不同服务商实例
- 出现问题时，优先以命令行请求验证 API 链路
