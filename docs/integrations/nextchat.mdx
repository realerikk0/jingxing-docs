# NextChat

NextChat 是一个轻量、自部署友好的聊天前端，适合通过 OpenAI 兼容方式接入井陉。对于井陉而言，推荐优先使用自部署环境变量方案完成接入，而不是在首轮接入时同时混用多个 provider。

## 适用场景

- 需要轻量级、自部署的聊天前端
- 需要通过 OpenAI 兼容接口接入井陉
- 需要快速建立聊天主链路

## 前置条件

请先准备以下信息：

- Base URL：`https://api.jingxing.uk/v1`
- API Key：专门用于 NextChat 的井陉令牌
- 模型名称：至少一个已经验证可用的模型

## 推荐接入方式

根据 NextChat 官方文档，自部署场景下最基础且稳定的接入方式为使用以下环境变量：

- `OPENAI_API_KEY`
- `BASE_URL`

井陉优先提供 OpenAI 兼容接口，因此该路径与 NextChat 的默认配置方式匹配度较高。

## 官方界面示意

以下界面示意来自 NextChat 官方仓库 README 的环境变量章节，用于说明自部署场景下最关键的配置项。

![NextChat 环境变量配置示意](/img/integrations/nextchat-env-doc.png)

与井陉接入直接相关的字段如下：

- `OPENAI_API_KEY`：填写井陉令牌
- `BASE_URL`：填写 `https://api.jingxing.uk/v1`
- 建议先完成最小变量组合验证，再继续扩展其他部署参数

## 配置步骤

### 1. 在井陉侧创建 NextChat 专用令牌

建议配置如下：

- 名称：如 `nextchat-web`
- 分组：适合 Web 聊天场景的通用分组
- 模型限制：仅开放计划在 NextChat 中使用的模型
- 额度：覆盖验证与日常使用需求

### 2. 在部署环境中设置变量

```bash
OPENAI_API_KEY=<YOUR_JINGXING_TOKEN>
BASE_URL=https://api.jingxing.uk/v1
```

无论使用 Docker、Vercel 还是 `.env.local`，本质上都是为 NextChat 提供以上最小配置。

### 3. 仅验证一个模型

部署完成后，建议仅选择一个已验证可用的模型执行首次对话请求，确认主链路稳定后再扩展更多模型。

## 可选的界面配置路径

若当前版本支持在界面中覆盖 OpenAI endpoint，也可以在设置中填写：

- API Key：井陉令牌
- Base URL：`https://api.jingxing.uk/v1`

但从维护角度看，自部署环境变量仍然是更推荐的长期方案。

## 验证方法

建议按以下顺序完成验证：

1. 在命令行中使用当前令牌执行 `GET /v1/models`
2. 配置 `OPENAI_API_KEY` 与 `BASE_URL`
3. 重启或重新部署 NextChat
4. 在界面中选择一个模型
5. 发起一次简单对话请求

## 常见问题

### 模型列表异常

建议优先检查：

- 当前令牌是否限制了模型
- 当前分组是否开放这些模型
- `BASE_URL` 是否准确指向 `https://api.jingxing.uk/v1`

### 认证失败

建议依次检查：

- `OPENAI_API_KEY` 是否完整复制
- 当前令牌是否过期、停用或无效
- NextChat 是否仍在使用旧环境变量

### 请求仍然发送到 OpenAI 官方地址

此类问题通常说明：

- `BASE_URL` 未生效
- 服务未重启
- 当前部署环境未正确加载变量

### 模型过多导致体验混乱

建议先保留少量模型，并明确默认模型，再逐步扩展。

## 实践建议

- 为 NextChat 单独签发令牌
- 优先使用环境变量接入井陉
- 首次仅配置少量模型
- 先验证主链路，再扩展其他配置
- 出现问题时，优先以命令行请求验证 API 链路
