# NextChat

如果你想自部署一个轻量、上手快、跨平台的聊天前端，并让底层模型统一走井陉，NextChat 是一个非常适合的选择。

和井陉配合时，最稳的路线不是先研究复杂多 provider，而是先把 **OpenAI 兼容路径** 跑通。

## 先说结论

- 想要最稳：优先用自部署的环境变量方式接井陉
- 想要最快：先只配 `OPENAI_API_KEY` 和 `BASE_URL`
- 想少踩坑：先只验证一个模型
- 想减少后续排障成本：给 NextChat 单独发令牌

## 接入前准备

先准备：

- Base URL：`https://api.jingxing.uk/v1`
- API Key：一个专门给 NextChat 使用的井陉令牌
- 模型名：至少一个已经验证可用的模型

## 为什么推荐先走 OpenAI 路径

根据 NextChat 官方仓库和文档，NextChat 自部署最基础、最稳定的接法就是通过：

- `OPENAI_API_KEY`
- `BASE_URL`

来配置 OpenAI 兼容接口。

井陉本身就优先提供 OpenAI 兼容接口，所以这条路径天然最顺。

## 推荐接入方式

### 1. 先在井陉创建一个 NextChat 专用令牌

推荐配置：

- 名称：`nextchat-web`
- 分组：通用聊天分组或适合 Web 前端的分组
- 模型限制：只开放你打算在 NextChat 中使用的模型
- 额度：先给足够验证的额度

### 2. 自部署时配置环境变量

最小可用配置通常是：

```bash
OPENAI_API_KEY=<YOUR_JINGXING_TOKEN>
BASE_URL=https://api.jingxing.uk/v1
```

如果你已经在用 Docker、Vercel 或 `.env.local`，本质上都是把这两项放进去。

### 3. 如果当前版本支持界面里的 custom endpoint

NextChat 文档在一些 provider 页面也展示了 custom endpoint 的配置方式。  
如果你当前版本支持在设置页覆盖 OpenAI endpoint，也可以在 UI 中填：

- API Key：井陉令牌
- OpenAI Endpoint / Base URL：`https://api.jingxing.uk/v1`

但长期看，**自部署环境变量** 仍然是更稳的主路径。

## 推荐模型选择思路

NextChat 更像通用聊天前端，所以优先级通常是：

1. 聊天稳定
2. 响应速度适中
3. 长对话可接受
4. 成本合理

如果只是验证接入，不要一开始就用高成本模型。

## 最短验证路径

1. 先用命令行跑通 `GET /models`
2. 配好 `OPENAI_API_KEY` 和 `BASE_URL`
3. 重启或重新部署 NextChat
4. 在 UI 中选择一个模型
5. 发起第一条最简单的对话

例如：

- “用一句话介绍井陉”
- “把这段文本总结成三点”

## 常见问题

### 页面能打开，但模型不对

优先检查：

- 当前 token 是否限制了模型
- 当前分组是否开放这些模型
- `BASE_URL` 是否真的指向 `https://api.jingxing.uk/v1`

### 一直 401

优先检查：

- `OPENAI_API_KEY` 是否复制完整
- token 是否过期或停用
- NextChat 是否还在使用旧环境变量

### 仍然在打 OpenAI 官方而不是井陉

这通常不是井陉问题，而是：

- `BASE_URL` 没生效
- 服务没有重启
- 当前部署环境没有正确读取变量

### 模型很多，但体验很乱

这一般不是“模型越多越好”，而是你没把用途和默认模型收好。  
更稳的做法是：

- 先只保留少量模型
- 先有一个清晰默认模型
- 先让主路径稳定，再逐步扩

## 推荐实践

- 给 NextChat 单独发令牌
- 优先用自部署环境变量接井陉
- 模型先少后多
- 先验证主链路，不要一上来混太多 provider
- 出问题先用命令行排除 API 侧问题
