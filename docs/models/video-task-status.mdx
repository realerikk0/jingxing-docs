# 视频任务状态查询

视频能力通常采用任务制处理流程，因此查询任务状态是视频接入中的关键环节。本文用于说明井陉当前两类视频任务的查询方式、状态含义和推荐轮询策略。

## 支持的查询路径

### OpenAI 兼容视频任务

当任务通过以下接口创建时：

- `POST /v1/videos`

后续应通过以下接口查询：

- `GET /v1/videos/{task_id}`
- `GET /v1/videos/{task_id}/content`

### Kling 视频任务

当任务通过以下接口创建时：

- `POST /kling/v1/videos/text2video`
- `POST /kling/v1/videos/image2video`

后续应通过以下接口查询：

- `GET /kling/v1/videos/text2video/{task_id}`
- `GET /kling/v1/videos/image2video/{task_id}`

## OpenAI 兼容任务查询

```bash
curl https://api.jingxing.uk/v1/videos/<TASK_ID> \
  -H "Authorization: Bearer <YOUR_TOKEN>"
```

常见返回字段包括：

- `id`
- `status`
- `progress`
- `created_at`
- `seconds`

任务完成后，可进一步获取视频内容：

```bash
curl -L https://api.jingxing.uk/v1/videos/<TASK_ID>/content \
  -H "Authorization: Bearer <YOUR_TOKEN>" \
  -o result.mp4
```

## Kling 任务查询

### 文生视频

```bash
curl https://api.jingxing.uk/kling/v1/videos/text2video/<TASK_ID> \
  -H "Authorization: Bearer <YOUR_TOKEN>"
```

### 图生视频

```bash
curl https://api.jingxing.uk/kling/v1/videos/image2video/<TASK_ID> \
  -H "Authorization: Bearer <YOUR_TOKEN>"
```

Kling 任务结果中，建议重点关注：

- `task_id`
- `status`
- `url`
- `format`
- `error`

## 状态字段说明

当前 OpenAPI 中明确出现的状态包括：

- `queued`
- `in_progress`
- `completed`
- `failed`

### `queued`

表示任务已接收，但仍在排队。

### `in_progress`

表示任务已经开始执行。

### `completed`

表示任务已完成。对于 OpenAI 兼容路径，应继续请求 `/content` 获取视频文件；对于 Kling 路径，通常应查看返回中的 `url`。

### `failed`

表示任务执行失败。建议优先区分是参数问题、权限问题还是上游链路异常，不宜直接原样重复提交。

## 轮询建议

推荐采用有限频率轮询策略：

1. 提交任务后先等待 3 到 5 秒
2. 随后每 5 到 10 秒查询一次
3. 设置总超时时间
4. 超时后再决定是否重新提交

该方式有助于降低不必要的轮询压力，并减少误判。

## 常见问题

### 混用 `task_id` 与 `id`

OpenAI 兼容接口通常返回 `id`，Kling 路径通常返回 `task_id`，应避免混用。

### 任务未完成即请求内容

此类情况通常会返回 404 或“文件不存在”。

### 查询路径错误

应先确认任务是通过 `/v1/videos` 创建，还是通过 `/kling/v1/videos/...` 创建，再选择对应查询路径。

### 持续轮询旧任务

排查时应确认当前查询的任务 ID 是否为本次提交返回的最新值。

### 失败后直接原样重提

如失败原因是请求参数或权限配置问题，原样重提通常不会解决问题。

## 使用建议

建议按以下顺序处理视频任务状态问题：

1. 确认任务创建路径
2. 确认查询路径
3. 确认任务 ID 正确
4. 在 `completed` 后再获取结果
