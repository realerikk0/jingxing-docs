# 视频任务状态查询

视频能力最容易卡住的一步，不是“怎么提交任务”，而是：

**任务已经提上去了，我到底该怎么判断它是在排队、执行、失败，还是已经能拿结果。**

这一页就是为这个场景准备的。

## 你会遇到两套查询方式

### OpenAI 兼容视频任务

适用于你先走了：

- `POST /v1/videos`

后续查询用：

- `GET /v1/videos/{task_id}`
- `GET /v1/videos/{task_id}/content`

### Kling 视频任务

适用于你走了：

- `POST /kling/v1/videos/text2video`
- `POST /kling/v1/videos/image2video`

后续查询用：

- `GET /kling/v1/videos/text2video/{task_id}`
- `GET /kling/v1/videos/image2video/{task_id}`

## OpenAI 兼容任务怎么查

```bash
curl https://api.jingxing.uk/v1/videos/<TASK_ID> \
  -H "Authorization: Bearer <YOUR_TOKEN>"
```

你通常会看到这些字段：

- `id`
- `status`
- `progress`
- `created_at`
- `seconds`

如果任务已经完成，再取内容：

```bash
curl -L https://api.jingxing.uk/v1/videos/<TASK_ID>/content \
  -H "Authorization: Bearer <YOUR_TOKEN>" \
  -o result.mp4
```

## Kling 任务怎么查

### 文生视频任务

```bash
curl https://api.jingxing.uk/kling/v1/videos/text2video/<TASK_ID> \
  -H "Authorization: Bearer <YOUR_TOKEN>"
```

### 图生视频任务

```bash
curl https://api.jingxing.uk/kling/v1/videos/image2video/<TASK_ID> \
  -H "Authorization: Bearer <YOUR_TOKEN>"
```

Kling 查询结果里，你通常最关心：

- `task_id`
- `status`
- `url`
- `format`
- `error`

## `status` 应该怎么理解

当前 OpenAPI 里明确出现的状态包括：

- `queued`
- `in_progress`
- `completed`
- `failed`

最稳的判断方式是：

### `queued`

任务已接收，但还在排队。

### `in_progress`

任务已经开始执行。

### `completed`

可以开始取结果了。  
OpenAI 兼容路径继续去拉 `/content`，Kling 路径通常直接看 `url`。

### `failed`

别先无脑重试。  
先看是参数问题、权限问题，还是上游波动。

## 推荐的轮询方式

不要每秒狂刷一次。  
运营和工程上都更稳的做法是：

1. 提交后先等 3 到 5 秒
2. 然后每 5 到 10 秒查一次
3. 设置总超时
4. 超时后再决定是否重新提交

## 最常见的 5 个错误

### 1. 把 `task_id` 和 `id` 混了

OpenAI 兼容接口更常返回 `id`，Kling 更常返回 `task_id`。  
别直接复制粘贴混用。

### 2. 任务还没完成就开始拿内容

这时最容易报 404 或“文件不存在”。

### 3. 查询路径写错

你要先确认自己是走的 `/v1/videos`，还是 `/kling/v1/videos/...`。  
两条线别混。

### 4. 一直轮询旧任务

调试时最容易发生：你以为自己在查新任务，实际上拿的还是上一次失败的 ID。

### 5. `failed` 后直接原样重发

如果失败原因是请求本身有问题，原样重发通常只会再失败一次。

## 推荐动作

如果你现在卡在“视频一直没结果”，按这个顺序最稳：

1. 先确认你提交的是哪种任务
2. 再确认查询路径
3. 再确认任务 ID 没拿错
4. 看到 `completed` 后再取内容
