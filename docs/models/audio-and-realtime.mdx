# 音频与 Realtime

井陉当前已提供文本转语音、音频转录、音频翻译与实时连接能力。本文用于说明不同能力对应的接口路径、最小请求和典型排查方向。

## 可用路径

当前仓库已明确暴露以下能力：

- `POST /v1/audio/speech`
- `POST /v1/audio/transcriptions`
- `POST /v1/audio/translations`
- `GET /v1/realtime`

## 能力划分

### 文本转语音

用于将文本生成音频文件，路径为：

- `POST /v1/audio/speech`

### 音频转录

用于将音频转换为原语言文本，路径为：

- `POST /v1/audio/transcriptions`

### 音频翻译

用于将音频翻译为英文文本，路径为：

- `POST /v1/audio/translations`

### Realtime

用于通过 WebSocket 建立实时对话或实时语音链路，路径为：

- `GET /v1/realtime`

## 最小请求：文本转语音

```bash
curl https://api.jingxing.uk/v1/audio/speech \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <YOUR_TOKEN>" \
  -d '{
    "model": "tts-1",
    "input": "欢迎使用井陉。",
    "voice": "alloy"
  }' \
  --output speech.mp3
```

该接口成功时通常直接返回音频流，而不是 JSON。

## 最小请求：音频转录

```bash
curl https://api.jingxing.uk/v1/audio/transcriptions \
  -H "Authorization: Bearer <YOUR_TOKEN>" \
  -F "file=@/absolute/path/to/sample.mp3" \
  -F "model=whisper-1" \
  -F "response_format=json"
```

首次验证建议使用以下输入条件：

- 短音频
- 清晰人声
- `response_format=json`

## 最小请求：音频翻译

```bash
curl https://api.jingxing.uk/v1/audio/translations \
  -H "Authorization: Bearer <YOUR_TOKEN>" \
  -F "file=@/absolute/path/to/sample.mp3" \
  -F "model=whisper-1"
```

如目标是普通转写，应优先使用 `transcriptions`，而非 `translations`。

## Realtime 连接说明

`/v1/realtime` 为 WebSocket 端点，不是普通的 HTTP JSON 接口。连接示例形式如下：

```text
wss://api.jingxing.uk/v1/realtime?model=gpt-4o-realtime-preview
```

建立连接时，应同时确认：

- 客户端使用的是 WebSocket 协议
- Bearer Token 已正确携带
- `model` 查询参数已填写

## 常见问题

### 将文本转语音接口误当作 JSON 接口处理

`/v1/audio/speech` 成功时返回音频流，调用侧应按文件或流式音频处理，而非 JSON 解析。

### 转录接口未使用 `multipart/form-data`

上传音频文件时，应确认请求体格式正确。

### 首次验证即使用较长音频

长音频会增加排查复杂度。建议先以短音频建立主链路。

### 混淆转写与翻译

保留原语言内容时应使用 `transcriptions`；需要翻译成英文时再使用 `translations`。

### 将 Realtime 当作普通 HTTPS 接口调用

Realtime 接口必须通过 WebSocket 方式建立连接。

### 聊天能力可用但音频能力不可用

此类问题通常与分组和模型权限有关，不应仅根据聊天能力状态作判断。

## 使用建议

建议按以下顺序验证音频相关能力：

1. 先完成文本转语音验证
2. 再完成短音频转录验证
3. 最后验证 Realtime 连接
