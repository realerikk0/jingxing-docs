# Embeddings

这一页解决的是：

**我想把文本转成向量，用于搜索、召回、相似度或 RAG，最小请求怎么发。**

当前项目里已经明确暴露了：

- `POST /v1/embeddings`
- `POST /v1/engines/{model}/embeddings`

实际开发里，优先用 `/v1/embeddings` 就够了。

## 最小请求

```bash
curl https://api.jingxing.uk/v1/embeddings \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <YOUR_TOKEN>" \
  -d '{
    "model": "text-embedding-3-large",
    "input": "井陉是一个统一接入多家 AI 模型的 API 网关。"
  }'
```

如果一切正常，返回里你最关心的是：

- `data[0].embedding`
- `model`
- `usage`

你第一次验证时，不需要立刻关心召回效果，先确认能稳定拿到向量。

## 这条接口最常见的 3 种用途

### 搜索 / 检索

把文档和查询都转成向量，再做相似度召回。

### RAG

先做 chunk，再做 embeddings，然后把召回结果送回聊天模型。

### 去重 / 聚类 / 相似判断

适合做语义近似、重复内容检测、内容归类。

## 哪些字段最值得先理解

### `model`

决定你用哪个向量模型。  
别先写死在程序里，先确认当前分组是否开放。

### `input`

可以是一段文本，也可以是一个文本数组。  
第一次验证建议只传一条字符串，不要先上批量。

### `dimensions`

当前 OpenAPI 里明确支持这个字段。  
如果你的向量库、现有索引或检索系统要求固定维度，再显式设置；否则第一次可以先不传。

### `encoding_format`

如果你的下游处理链路对格式有要求，再加；第一次打通链路通常不需要先管它。

## 最稳的验证顺序

1. 先拿一条短文本做单条嵌入
2. 确认返回里真的有向量数组
3. 再试两条文本批量嵌入
4. 最后再决定是否指定 `dimensions`

## 最常见的 5 个问题

### 1. 当前分组只开了聊天模型，没开 embeddings

这是最常见的权限问题之一。

### 2. 一上来就批量几十条

第一次排错时这样做没有必要。  
单条先通，排查成本最低。

### 3. `input` 结构写乱

有些 SDK 里会帮你包一层结构，换到 `curl` 或自定义请求时容易出错。

### 4. 指定了错误维度

如果你的下游索引和当前模型维度不一致，后面会很痛苦。  
所以第一次最好先看原始输出。

### 5. 只验证生成，不验证消费

很多人拿到向量就停了，但真正的问题常常出在“向量库写入 / 查询 / 维度匹配”这一步。

## 推荐动作

如果你现在就要开始做向量检索：

1. 先跑一条最小 `embeddings`
2. 把结果写进你的向量库
3. 再做一条查询向量
4. 最后才开始调 chunk 策略
