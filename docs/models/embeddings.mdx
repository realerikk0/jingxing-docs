# Embeddings

井陉当前已提供向量化接口，用于搜索、召回、语义相似度计算和 RAG 场景。本文用于说明最小请求、参数含义与常见验证方法。

## 可用路径

当前仓库已明确暴露以下路径：

- `POST /v1/embeddings`
- `POST /v1/engines/{model}/embeddings`

在实际接入中，通常优先使用 `/v1/embeddings` 即可。

## 最小请求

```bash
curl https://api.jingxing.uk/v1/embeddings \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <YOUR_TOKEN>" \
  -d '{
    "model": "text-embedding-3-large",
    "input": "井陉是一个统一接入多家 AI 模型的 API 网关。"
  }'
```

验证时建议重点确认以下字段：

- `data[0].embedding`
- `model`
- `usage`

## 典型用途

### 搜索 / 检索

将文档与查询统一转换为向量，并用于相似度召回。

### RAG

将切分后的文本执行向量化，再将召回结果交由聊天模型生成最终响应。

### 去重 / 聚类 / 相似判断

用于语义近似、内容归类与重复检测。

## 参数说明

### `model`

用于指定向量模型。建议在调用前先确认当前分组是否开放目标模型。

### `input`

可为单条文本，也可为文本数组。首次验证建议优先使用单条文本，以降低排查复杂度。

### `dimensions`

当前 OpenAPI 明确支持该字段。如向量库或下游检索系统要求固定维度，可显式设置；如无明确约束，可在初次验证阶段省略。

### `encoding_format`

如当前下游链路对编码格式有明确要求，可显式设置；首次打通链路时通常无需优先关注。

## 建议验证顺序

建议按以下顺序建立向量化主链路：

1. 先使用单条短文本执行嵌入
2. 确认返回中存在向量数组
3. 再尝试小规模批量嵌入
4. 最后再评估是否需要显式指定 `dimensions`

## 常见问题

### 当前分组未开放向量模型

这是最常见的权限问题之一。若聊天模型可用但向量接口不可用，应优先检查分组和模型开放情况。

### 首次即执行大批量请求

大批量请求会显著提高排查复杂度。建议先以单条请求完成主链路验证。

### `input` 结构错误

在 SDK 与自定义请求之间切换时，`input` 的实际结构容易被误写，应优先核对请求体。

### 维度与下游索引不匹配

如显式指定了错误维度，后续写入向量库或执行查询时可能出现不兼容问题。

### 仅验证生成，不验证消费链路

获得向量并不代表检索链路已完全可用，还应继续验证向量写入、查询和维度匹配情况。

## 使用建议

建议在首轮接入中先完成以下事项：

1. 验证单条向量生成
2. 验证向量写入向量库
3. 验证查询向量生成
4. 最后再开始调优 chunk 与召回策略
