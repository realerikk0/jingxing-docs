# 文生图

井陉当前已提供 `POST /v1/images/generations` 接口，用于执行图片生成请求。当前 OpenAPI 示例使用 `qwen-image-plus`，实际可用模型仍以当前分组与令牌权限为准。

## 适用场景

- 验证图片生成能力是否已开放
- 为后端服务或客户端建立首条图片请求链路
- 将图片生成问题与客户端问题分离排查

## 前置条件

开始前，请确认以下条件已经满足：

- 当前令牌可用
- 当前分组已开放至少一个图片模型
- Base URL 为 `https://api.jingxing.uk/v1`
- 已准备一段结构清晰的提示词

## 最小请求

```bash
curl https://api.jingxing.uk/v1/images/generations \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <YOUR_TOKEN>" \
  -d '{
    "model": "qwen-image-plus",
    "input": {
      "messages": [
        {
          "role": "user",
          "content": [
            {
              "text": "一只戴着红围巾的白猫坐在木窗边，清晨阳光打进来，电影感摄影，细节丰富"
            }
          ]
        }
      ]
    },
    "parameters": {
      "size": "1328*1328",
      "prompt_extend": true,
      "watermark": false,
      "negative_prompt": ""
    }
  }'
```

## 返回结果说明

成功时，返回结果通常包含：

- `created`
- `data[].url`
- `data[].revised_prompt`

验证时建议重点确认：

- `data` 中是否包含图片结果
- `revised_prompt` 是否偏离原始意图过大

## 常用参数说明

### `model`

用于指定图片模型。建议先通过 `GET /v1/models` 或控制台确认模型可见性，再填写目标值。

### `input.messages`

该接口当前使用消息数组结构，而不是单独的顶层 `prompt` 字段。首次调用建议使用最简单结构：

- `role` 固定为 `user`
- `content` 中仅包含一个 `text`

### `parameters.size`

用于控制输出尺寸。首次验证建议使用常规尺寸，不建议在首轮测试时直接使用极端分辨率。

### `parameters.prompt_extend`

用于控制系统是否扩展提示词。首次验证可开启，以获得较稳定的出图效果；如需严格对比提示词效果，可在后续测试中关闭。

### `parameters.watermark`

用于控制是否添加水印。内部测试场景通常会先关闭。

## 建议排查顺序

如图片生成失败，建议按以下顺序排查：

1. 确认令牌有效
2. 确认模型在当前分组中可见
3. 缩短提示词
4. 将尺寸调整为常规值
5. 最后再考虑上游波动

## 常见问题

### 模型不可见

通常说明当前分组未开放目标模型，而非接口本身异常。

### 使用了错误的请求结构

该接口当前使用 `input.messages` 结构。若直接使用顶层 `prompt`，请求通常无法成功。

### 提示词过于空泛

过于简单的提示词即使能够出图，也往往难以获得稳定结果。建议明确描述主体、环境、风格和画面方向。

### 首次即使用大尺寸

大尺寸会增加验证成本，不利于首轮排查。建议先以链路建立为目标，再逐步扩展参数。

### `curl` 正常但客户端失败

此类情况通常说明问题更可能出在客户端请求封装，而非网关主链路。

## 使用建议

建议按以下顺序建立图片主链路：

1. 执行 `GET /v1/models`
2. 确认一个图片模型
3. 发送最小图片请求
4. 成功后再调整尺寸与提示词
