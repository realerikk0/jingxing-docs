# Kling 文生视频

井陉当前已提供 Kling 风格的视频生成接口，用于直接提交文生视频任务，并通过对应状态接口获取结果。本文用于说明该接口的最小请求、参数含义与验证方法。

## 可用路径

当前仓库已明确暴露以下路径：

- `POST /kling/v1/videos/text2video`
- `GET /kling/v1/videos/text2video/{task_id}`

## 最小请求

```bash
curl https://api.jingxing.uk/kling/v1/videos/text2video \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <YOUR_TOKEN>" \
  -d '{
    "model": "kling-v1",
    "prompt": "宇航员站起身向前走，尘土轻轻扬起，镜头缓慢推进",
    "duration": 5,
    "width": 1280,
    "height": 720,
    "fps": 30
  }'
```

成功后通常会先返回任务信息：

```json
{
  "task_id": "abcd1234efgh",
  "status": "queued"
}
```

## 状态查询

```bash
curl https://api.jingxing.uk/kling/v1/videos/text2video/<TASK_ID> \
  -H "Authorization: Bearer <YOUR_TOKEN>"
```

建议重点关注以下字段：

- `status`
- `url`
- `error`

## 参数说明

### `model`

OpenAPI 当前已明确示例模型包括 `kling-v1` 等。实际可用模型仍取决于当前分组开放情况。

### `prompt`

建议在首次验证时使用结构清晰、动作单一的提示词，避免在一条提示中叠加过多角色、动作或镜头要求。

### `duration`

建议在首次验证时使用较短时长，以降低任务失败时的排查成本。

### `width` / `height`

建议优先使用常规尺寸，在验证通过后再逐步扩展规格。

## 常见问题

### 提示词过于复杂

过多主体、动作和镜头要求会降低首轮验证的稳定性。

### 任务尚未完成即尝试读取结果

应先完成状态查询，确认任务完成后再处理最终结果。

### 当前令牌未开放 Kling 模型

此类问题通常与分组策略有关，应先确认模型是否在当前令牌可见范围内。

### 将该接口理解为同步接口

该接口属于任务制接口，不会立即返回完整视频内容。

## 使用建议

建议按以下顺序验证文生视频链路：

1. 使用短提示词提交任务
2. 使用较短时长完成首轮验证
3. 获取 `task_id`
4. 查询状态并在成功后再扩大规格
