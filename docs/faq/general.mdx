# 通用问题

本页汇总最常见的基础认知与上手类问题。  
如需排查错误、计费或安全问题，建议直接进入对应 FAQ 页面。

## 井陉是什么

井陉是一个统一制式的 AI 网关与工作台。  
它将多上游模型、分组策略、令牌管理、额度控制与审计能力集中在同一层，并尽量对外保持 OpenAI 兼容调用方式。

可以从以下三个角色理解井陉：

- 对开发者而言，井陉是统一 API 入口
- 对管理员而言，井陉是统一模型、令牌与额度管理层
- 对客户端而言，井陉是更易替换与治理的网关入口

## 井陉适合谁用

井陉主要面向以下三类使用者：

- 需要以统一接口接入多个模型的开发者
- 需要控制团队成本、权限与日志的管理员
- 需要将 Cursor、Cherry Studio、Lobe Chat、Open WebUI 等客户端统一接入一处的团队

## 为什么要使用分组

可用模型范围、价格、稳定性目标与额度策略通常不应对所有用户完全一致。

分组的主要价值包括：

- 将测试流量与生产流量分离
- 将低成本场景与高稳定场景分离
- 将不同团队与不同用途分离

缺少分组后，权限治理、成本控制与日志审计通常会迅速变得复杂。

## API 和网页工作台有什么区别

API 与网页工作台并非替代关系，而是承担不同职责：

- API：面向程序、脚本、SDK 与第三方客户端
- 网页工作台：面向令牌、分组、额度、渠道与日志管理

可以概括为：

- 使用井陉时，主要依赖 API
- 管理井陉时，主要依赖网页工作台

## 井陉是否只适合程序员

不是。  
虽然井陉本质上是 API 网关，但只要客户端支持 OpenAI 兼容接口，非程序员也可以通过桌面或 Web 客户端使用。

对于非开发者，建议优先从客户端接入页面开始，而非直接阅读 API 参考。

## 为什么文档一直强调“单独发令牌”

因为这是最直接且最有效的治理方式之一。

令牌混用于过多场景后，通常很难判断：

- 哪个客户端在消耗额度
- 哪个团队正在使用高成本模型
- 哪个工具正在持续报错

在令牌按用途拆分后，排查与审计成本会显著降低。

## 为什么文档建议先用 `GET /models` 验证

因为这是最轻量、也最适合快速分层定位问题的验证方式之一。  
如 `GET /models` 无法通过，通常无需优先怀疑复杂客户端行为。

该接口至少可以验证：

- 域名是否可达
- 令牌是否有效
- Base URL 是否正确
- 当前令牌大致可以看到哪些模型

## 应该先接客户端，还是先接 API

如目标是尽快开始使用，客户端接入通常更直接。  
如目标是服务集成、自动化或团队级部署，建议优先完成 API 验证。

推荐顺序如下：

1. 先完成最小 API 请求
2. 再接入客户端
3. 最后建立团队级规则与多分组治理
