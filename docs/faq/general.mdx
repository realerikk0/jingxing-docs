# 通用问题

这一页聚焦最常见的“先弄明白它是什么、适合怎么用、从哪里开始”的问题。  
如果你已经在排查错误、计费或安全问题，可以直接跳到对应 FAQ。

## 井陉是什么

井陉是一个统一制式的 AI 网关与工作台。  
它把多上游模型、多分组策略、令牌管理、额度与审计放到同一层里，对外则尽量保持一致的 OpenAI 兼容调用方式。

你可以把它理解成：

- 对开发者来说，是统一 API 入口
- 对管理员来说，是统一模型、令牌和额度管理层
- 对客户端来说，是一个更容易替换接入点的网关

## 井陉适合谁用

最适合三类人：

- 想用统一接口接多个模型的开发者
- 想控制团队成本、权限和日志的管理员
- 想把 Cursor、Cherry Studio、Lobe Chat、Open WebUI 等客户端统一接到一处的人

## 为什么要用分组

因为“可用模型范围、价格、稳定性、额度策略”通常不应该对所有人都完全一样。

分组的价值在于：

- 把测试流量和生产流量分开
- 把低成本场景和高稳定场景分开
- 把不同团队和不同用途分开

如果没有分组，后面基本一定会遇到权限难管、成本难控和日志难查的问题。

## API 和网页工作台有什么区别

它们不是互相替代，而是分工不同。

- API：给程序、脚本、SDK 和第三方客户端使用
- 网页工作台：给人来管理令牌、分组、额度、渠道和日志

一句话理解：

- 你“用”井陉时主要靠 API
- 你“管”井陉时主要靠网页工作台

## 井陉是不是只适合程序员

不是。  
虽然它本质上是 API 网关，但只要客户端支持 OpenAI 兼容接口，非程序员也可以通过桌面或 Web 客户端来使用。

对于非开发者，更推荐从客户端页面开始，而不是直接看 API 参考。

## 为什么文档一直强调“单独发令牌”

因为这是最便宜也最有效的治理方式之一。

同一个令牌混给太多场景后，你会很难判断：

- 哪个客户端在消耗额度
- 哪个团队在用高成本模型
- 哪个工具在报错

而单独发令牌后，排查和审计都轻很多。

## 为什么文档里总说“先用 GET /models 验证”

因为它是最轻量、最能快速分层定位问题的检查方式之一。  
如果 `GET /models` 都不通，就没必要先去怀疑复杂客户端。

它至少能验证：

- 域名是否可达
- token 是否有效
- Base URL 是否正确
- 当前令牌大致能看到哪些模型

## 我应该先接客户端，还是先接 API

如果你只是想尽快开始用，先接客户端更轻松。  
如果你要做服务集成、自动化或团队级接入，还是建议先把 API 跑通。

最稳的顺序通常是：

1. 先跑最小 API 请求
2. 再接客户端
3. 最后再做团队级规则和多分组治理
