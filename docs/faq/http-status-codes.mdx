# HTTP 状态码总览

这一页是给你快速查错用的。  
如果你只想知道“现在这次报错大概要先查哪里”，优先看这里；如果你想看更完整的重试策略，再去看[错误与重试](/docs/platform/errors-and-retries)。

## 一张表先看懂

| 状态码 | 最常见含义 | 第一反应该做什么 | 要不要自动重试 |
| --- | --- | --- | --- |
| 400 | 参数错误、请求体错误、模型名错误 | 先改请求 | 不要 |
| 401 | 没带 token、token 无效、token 过期 | 先查鉴权 | 不要 |
| 403 | 有 token，但权限不够 | 先查分组 / 模型权限 / 来源限制 | 不要 |
| 404 | 地址错了、路径错了、模型不可见 | 先查 URL 和模型名 | 不要 |
| 429 | 频率限制、额度限制、上游限流 | 降速、退避、查分组 | 可以有限重试 |
| 500 | 平台或上游内部错误 | 缩小请求、查日志 | 视情况 |
| 502 | 网关或上游异常 | 退避重试、换模型 | 可以有限重试 |
| 503 | 服务暂不可用 | 等待恢复或换分组 | 可以有限重试 |
| 504 | 上游超时 | 缩短上下文、换更稳分组 | 可以有限重试 |

## 400 参数错误

最常见原因：

- `model` 写错
- `messages` 结构不合法
- JSON 格式本身有问题
- 客户端自动带了当前模型不支持的参数

最短动作：

1. 先检查请求体
2. 再确认模型名
3. 再用最小示例重发一次

## 401 未认证

最常见原因：

- 没带 `Authorization: Bearer <TOKEN>`
- token 复制不完整
- token 已过期
- token 已停用

最短动作：

1. 先跑 `GET /models`
2. 确认请求头里有 Bearer Token
3. 确认你用的是当前新令牌，不是旧缓存

## 403 无权限

最常见原因：

- 当前分组不开放这个模型
- 令牌本身限制了模型范围
- 开了 IP / 来源限制

这类报错通常不适合重试，因为问题不在“波动”，而在“规则本身不允许”。

## 404 找不到

井陉里最常见的 404 不是系统消失了，而是：

- 你把地址写成了 `https://api.jingxing.uk`
- 而不是 `https://api.jingxing.uk/v1`

也有一类情况是模型在当前 token 下根本不可见，客户端把它解释成“找不到”。

## 429 频率限制

最常见原因：

- 同一个客户端打得太快
- 多个客户端共用同一个令牌
- 当前分组更偏低成本，不适合高并发
- 上游模型当时在限流

最短动作：

1. 先降低频率
2. 再做指数退避
3. 再考虑换更稳的分组或模型

## 500 内部错误

这是最容易让人误判的一类。

它可能表示：

- 平台内部处理失败
- 上游返回了通用失败
- 当前请求复杂度过高

最短动作：

1. 先缩短上下文
2. 再换一个更简单的测试请求
3. 看是否只在某个模型或某个客户端下复现

## 502 / 503 / 504

这三类更像链路问题，不像配置问题。

### 502

通常是网关或上游异常。

### 503

通常是服务暂不可用。

### 504

通常是上游响应太慢或超时。

这三类问题最适合：

- 有上限的自动重试
- 指数退避
- 切换更稳的分组或更轻的模型

## 最短排查顺序

不管你遇到哪一种状态码，最稳的排查顺序都是：

1. 先看状态码本身
2. 再跑 `GET /models`
3. 再发一条最小 `chat/completions`
4. 最后再回具体客户端排查

## 客户端用户特别容易踩的坑

### Cursor

- Base URL 忘了写 `/v1`
- 还在用旧 token
- 开了 override 但模型没切对

### Cherry Studio

- 服务商建对了，但模型没手动加
- Key 正确，但当前分组不开放该模型

### Lobe Chat / Open WebUI

- 环境变量改了，但服务没重启
- 页面打开正常，被误以为 API 一定也正常

## 推荐动作

如果你只是想快速恢复使用，不要一次同时排 5 件事。  
按这个顺序来最省时间：

1. 先确认 token
2. 先确认 Base URL
3. 先确认模型名
4. 再确认分组
5. 最后再看是不是链路波动
