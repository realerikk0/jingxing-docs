# HTTP 状态码总览

本文用于提供高频状态码的快速判读方式，帮助在较短时间内确定问题属于参数、鉴权、权限、限流还是上游链路异常。

如需查看更完整的重试策略与排障方法，建议继续阅读[错误与重试](/docs/platform/errors-and-retries)。

## 状态码速查表

| 状态码 | 常见含义 | 首要排查方向 | 是否适合自动重试 |
| --- | --- | --- | --- |
| 400 | 参数错误、请求结构错误、模型名错误 | 检查请求体与模型名 | 否 |
| 401 | 未认证、令牌无效、令牌过期 | 检查鉴权头与令牌状态 | 否 |
| 403 | 有令牌但无权限 | 检查分组、模型权限、来源限制 | 否 |
| 404 | 地址错误、模型不可见、路径错误 | 检查 URL 与模型可见性 | 否 |
| 429 | 频率限制、额度限制、上游限流 | 降低频率并检查分组 | 有限重试 |
| 500 | 平台内部错误或上游异常 | 缩小请求并检查日志 | 视情况 |
| 502 | 网关或上游异常 | 退避重试或切换模型 | 有限重试 |
| 503 | 服务暂时不可用 | 等待恢复或切换分组 | 有限重试 |
| 504 | 上游超时 | 缩短上下文或切换更稳定分组 | 有限重试 |

## 400 参数错误

常见原因包括：

- `model` 字段错误
- `messages` 结构不合法
- JSON 格式不完整
- 客户端自动附加了当前模型不支持的参数

建议排查顺序：

1. 检查请求体格式
2. 检查模型名称
3. 使用最小请求重新验证

## 401 未认证

常见原因包括：

- 未携带 `Authorization: Bearer <TOKEN>`
- 令牌复制不完整
- 令牌已过期
- 令牌已停用

建议排查顺序：

1. 执行 `GET /v1/models`
2. 检查请求头中的 Bearer Token
3. 确认当前使用的是否为最新令牌

## 403 无权限

常见原因包括：

- 当前分组未开放目标模型
- 令牌设置了模型限制
- 已启用 IP 或来源限制

此类问题通常不适合通过重试解决，应优先检查权限配置本身。

## 404 找不到

常见原因包括：

- 接口地址未包含 `/v1`
- 请求路径错误
- 模型在当前令牌下不可见

建议优先核对：

- 是否使用 `https://api.jingxing.uk/v1`
- 是否填写了正确模型名称

## 429 频率限制

常见原因包括：

- 单个客户端请求频率过高
- 多个客户端共用同一令牌
- 当前分组不适合高并发
- 上游模型处于限流状态

建议处理方式：

1. 降低请求频率
2. 启用指数退避
3. 必要时切换更稳定的分组或模型

## 500 / 502 / 503 / 504

此类状态码通常更接近平台内部或上游链路问题。

### 500

可能表示平台内部处理失败、上游返回通用错误，或当前请求复杂度过高。

### 502

通常表示网关或上游服务异常。

### 503

通常表示服务暂时不可用。

### 504

通常表示上游响应超时。

建议处理方式：

- 使用更小的测试请求复现
- 保持有限次数重试
- 必要时切换模型或分组

## 建议排查顺序

对于大多数状态码，建议采用以下统一顺序：

1. 先确认状态码类别
2. 再执行 `GET /v1/models`
3. 再发送最小 `chat/completions` 请求
4. 最后回到具体客户端环境排查

## 常见客户端排查重点

### Cursor

- Base URL 是否遗漏 `/v1`
- 是否仍在使用旧令牌
- 当前模型是否已正确切换

### Cherry Studio

- 自定义服务商配置是否正确
- 模型是否已手动添加
- 当前分组是否开放目标模型

### Lobe Chat / Open WebUI

- 环境变量修改后是否已重启服务
- 页面可访问是否被误判为 API 一定可用

## 使用建议

状态码排查应尽量避免同时修改多个变量。建议优先确认：

1. 令牌
2. Base URL
3. 模型名
4. 分组权限
5. 是否存在链路波动
