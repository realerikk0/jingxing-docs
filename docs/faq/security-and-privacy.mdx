# 安全与隐私 FAQ

本页汇总最常见的安全与隐私问题。  
涉及日志、存储与可见范围时，应始终以当前平台配置、控制台展示与管理员策略为准。

## 请求内容是否会被保存

不应默认理解为“完全不保存任何信息”，也不应默认理解为“所有正文均长期留存”。  
通常应按以下原则理解：

- 平台通常至少会保留必要的调用元信息，例如时间、模型、令牌、状态码与消耗
- 是否保存请求正文、响应正文以及保存时长，取决于当前平台配置与日志策略

如业务存在明确合规要求，应直接依据平台当前说明与管理员配置判断。

## 谁可以查看调用日志

通常并非所有用户都能查看全部日志。常见权限模型如下：

- 普通用户仅能查看与自身相关的日志
- 管理员可查看更高层级的调用信息
- 是否可见请求正文与响应正文，仍取决于当前日志策略

“管理员可查看日志”并不等同于“所有内容一定可见”，两者不应混同。

## 如何安全地向团队成员分发令牌

不推荐采用以下方式分发令牌：

- 发送到群聊
- 放在长期不清理的在线文档中
- 直接写入代码仓库

更推荐采用以下方式：

- 为每个成员或每个客户端单独签发令牌
- 通过密钥管理工具或安全环境变量传递
- 如需临时发送，发送后尽快轮换

## 令牌泄露后应如何处理

建议按以下顺序处理：

1. 先停用或轮换可疑令牌
2. 再检查最近的调用日志与消耗
3. 再决定是否需要收紧模型、分组或来源限制

如已怀疑令牌泄露，通常应优先完成止损，再进行详细分析。

## 是否支持 IP 白名单或来源限制

该类能力通常属于平台级或令牌级附加安全策略。是否启用、如何启用以及在哪一层启用，取决于当前平台配置。

以下场景通常适合增加来源限制：

- 后端服务
- 固定服务器出口
- CI / 自动任务

个人本地开发则不宜在初始阶段设置过严限制，以免在切换网络时造成误伤。

## 为什么文档一直强调“单独发令牌”

因为这不仅是运维习惯，也是最重要的安全边界之一。

单独发令牌至少有以下好处：

- 泄露时影响范围更小
- 日志更易追踪
- 额度与模型权限更易收口

## 如何更安全地让客户端使用令牌

建议按以下顺序配置：

1. 为每个客户端单独签发令牌
2. 仅开放该客户端实际需要的模型
3. 根据需要设置过期时间
4. 对服务端或固定出口增加来源限制

该方式可有效降低单个客户端配置泄露对整体平台造成的影响。

## `401`、`403` 与安全问题有什么关系

两者都与鉴权或权限有关，但含义不同：

- `401` 更常见于 token 缺失、无效或过期
- `403` 更常见于分组、模型权限或来源限制不允许

因此：

- `401` 应先检查 token
- `403` 应先检查策略

## 是否建议长期保留令牌

不建议将所有令牌设置为长期有效。推荐做法如下：

- 临时测试：短有效期
- 团队客户端：定期轮换
- 生产服务：使用独立令牌并配套轮换流程

## 基本原则

如业务对安全与隐私要求较高，最需要避免的并非功能不足，而是依赖未经验证的默认假设。  
更稳妥的做法是：

- 明确用途
- 缩小权限
- 单独签发令牌
- 及时轮换
- 以平台当前配置为准
