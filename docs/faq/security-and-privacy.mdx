# 安全与隐私 FAQ

这一页只回答最常见的安全与隐私问题。  
有一点需要先说明：**涉及日志、存储和可见范围时，以当前平台实际配置、控制台展示和管理员策略为准。**

## 请求内容是否会被保存

不要默认理解成“绝对不保存任何东西”，也不要默认理解成“所有正文都会长期留存”。  
更稳的理解方式是：

- 平台通常至少会保留必要的调用元信息
  比如时间、模型、令牌、状态码、消耗等
- 是否保存请求正文、响应正文、保存多久，取决于当前平台配置和日志策略

如果你对这件事有明确合规要求，不要靠猜，直接以平台当前说明和管理员配置为准。

## 谁能看到调用日志

通常不是所有用户都能看到全部日志。  
更常见的情况是：

- 普通用户只能看到自己相关的部分
- 管理员能看到更高层级的调用信息
- 是否能看到请求正文、响应正文，也取决于当前日志策略

所以不要把“管理员能看到日志”和“所有内容一定都可见”混为一谈。

## 怎样安全地把令牌发给团队成员

最不推荐的做法是：

- 发在群聊里
- 发在长期不清理的在线文档里
- 直接写进代码仓库

更推荐：

- 给每个成员或每个客户端单独发令牌
- 用密钥管理工具或安全环境变量传递
- 如果必须临时发送，发送后尽快轮换

## 泄露令牌后怎么办

最稳的顺序是：

1. 先停用或轮换可疑令牌
2. 再看最近的调用日志和消耗
3. 再决定是否需要收紧模型、分组或来源限制

不要先花很多时间分析，再决定停不停。  
如果你已经怀疑令牌泄露，先止损通常更重要。

## 是否支持 IP 白名单或来源限制

这类能力通常是平台级或令牌级的附加安全策略。  
是否启用、如何启用、在哪一层启用，取决于当前平台配置。

如果你的场景是：

- 后端服务
- 固定服务器出口
- CI / 自动任务

那非常适合加来源限制。  
如果是个人本地开发，就不要一上来绑得太死，否则自己换网络时最容易误伤。

## 为什么文档一直强调“单独发令牌”

因为它不仅仅是运维习惯，也是安全边界。

一个令牌只给一个用途，至少有三个好处：

- 泄露时影响面更小
- 日志更容易追踪
- 额度和模型权限更容易收口

## 如何更安全地给客户端使用令牌

推荐顺序是：

1. 给每个客户端单独发令牌
2. 只开放它真正要用的模型
3. 按需要设置过期时间
4. 对服务端或固定出口加来源限制

这样即使某个客户端配置泄露，也不会把整个平台的风险一起放大。

## 401、403 和安全有什么关系

它们经常长得像“都是权限问题”，但本质不同：

- `401` 更常见于 token 本身无效、缺失或过期
- `403` 更常见于分组、模型权限、来源限制不允许

所以排障时：

- `401` 先查 token
- `403` 先查策略

## 我需要长期保留令牌吗

不建议把所有令牌都长期有效。  
更稳的做法是：

- 临时测试：短有效期
- 团队客户端：定期轮换
- 生产服务：单独令牌并配轮换流程

## 最后一个原则

如果你对安全和隐私的要求比较高，最危险的不是“功能不够多”，而是“默认假设太多”。  
运营型文档里最稳的做法永远是：

- 明确用途
- 缩小权限
- 单独发令牌
- 及时轮换
- 以平台当前配置为准
