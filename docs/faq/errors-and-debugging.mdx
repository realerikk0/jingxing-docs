# 报错与排查 FAQ

这一页不展开讲完整机制，而是把最常遇到的问题直接给到最短答案。  
如果你想系统理解状态码、重试和排查顺序，优先看[错误与重试](/docs/platform/errors-and-retries)。

## 为什么提示未提供令牌

最常见原因只有两个：

- 没带 `Authorization: Bearer <TOKEN>`
- 带了请求头，但 token 为空或复制不完整

最快验证方法是先用命令行跑一次：

```bash
curl https://api.jingxing.uk/v1/models \
  -H "Authorization: Bearer <YOUR_TOKEN>"
```

如果这条都不通，先不要怀疑客户端，先把最小请求跑通。

## 为什么提示模型不存在

通常不是“平台里完全没有这个模型”，而是下面几种情况：

- 模型名写错
- 当前分组不开放这个模型
- 当前令牌限制了模型范围
- 客户端缓存了旧模型列表

最短排查方式：

1. 先看 `GET /models`
2. 再看当前客户端选择的模型名
3. 再看当前令牌和分组

## 为什么同一个请求有时成功有时失败

最常见原因不是“系统随机抽风”，而是请求链路里有波动：

- 当前模型或上游临时承压
- 客户端自动重试
- 分组策略更偏低成本而不是高稳定性
- 上下文长度在某些时候明显更大

如果是这种“偶发成功 / 偶发失败”，优先看：

- 是否集中在某个模型
- 是否集中在某个客户端
- 是否集中在某个时间段

## 为什么客户端一直超时

先别急着换 token，优先看这几项：

- Base URL 是否正确
- 当前模型是否适合长上下文或交互式场景
- 当前分组是否更偏低成本
- 当前对话是否已经过长

如果是开发工具类客户端，还要额外怀疑：

- 它是否偷偷重试
- 它是否默认附带了当前模型不支持的参数
- 它是否缓存了旧配置

## 为什么控制台能登录，但 API 不可用

这是两个完全不同的链路。

- 控制台可用，只能说明前台登录和页面访问没问题
- API 可用，还要求令牌、Base URL、请求头和模型权限都正确

最常见的误区是：看到控制台正常，就默认 API 也一定正常。  
实际上 API 是否可用，还是要靠最小请求单独验证。

## 为什么 401 和 403 看起来很像

它们都像“权限问题”，但本质不一样：

- `401` 更像“你没正确证明自己是谁”
- `403` 更像“我知道你是谁，但你不能这么做”

所以：

- `401` 先查 token 和请求头
- `403` 先查模型权限、分组和来源限制

## 为什么 429 一直出现

最常见原因：

- 同一个客户端短时间打太快
- 多个客户端共用同一个令牌
- 当前分组更适合低成本而不是高并发
- 上游本身在限流

最有效的改善方式通常不是“无脑多重试几次”，而是：

- 做指数退避
- 给不同客户端拆令牌
- 切更适合当前场景的分组

## 为什么我换了新令牌还是不行

这说明问题未必在令牌本身。

还要继续检查：

- Base URL 是否写对
- 客户端是否仍缓存旧配置
- 当前模型是不是仍然写错
- 当前客户端接法本身是否正确

## 哪些报错适合重试

一般来说：

- 适合重试：`429`、`502`、`503`、`504`
- 不适合重试：`400`、`401`、`403`、`404`

如果你已经知道是参数错了或权限不够，继续重试只会浪费时间和额度。

## 遇到报错时最短动作

1. 先看状态码
2. 先跑 `GET /models`
3. 再发一条最小 `chat/completions`
4. 再回到具体客户端排查
