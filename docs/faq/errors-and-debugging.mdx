# 报错与排查 FAQ

本页以速查方式整理高频报错问题。  
如需系统理解状态码、重试原则与排查顺序，建议优先阅读[错误与重试](/docs/platform/errors-and-retries)。

## 为什么提示未提供令牌

最常见原因包括：

- 未携带 `Authorization: Bearer <TOKEN>`
- 已携带请求头，但 token 为空或复制不完整

最快的验证方式是先执行一次命令行请求：

```bash
curl https://api.jingxing.uk/v1/models \
  -H "Authorization: Bearer <YOUR_TOKEN>"
```

如该请求无法通过，应优先完成最小请求验证，而非直接排查客户端。

## 为什么提示模型不存在

通常并非平台不存在该模型，而是出现以下情况之一：

- 模型名填写错误
- 当前分组未开放该模型
- 当前令牌限制了模型范围
- 客户端缓存了旧模型列表

建议按以下顺序排查：

1. 先查看 `GET /models`
2. 再核对客户端当前选中的模型名
3. 再确认当前令牌与分组配置

## 为什么同一个请求有时成功，有时失败

该类现象通常与请求链路波动有关，常见原因包括：

- 当前模型或上游短时承压
- 客户端自动重试
- 分组策略偏向低成本而非高稳定性
- 上下文长度在不同请求之间显著变化

如属于偶发成功、偶发失败的现象，建议优先观察：

- 是否集中发生在某个模型
- 是否集中发生在某个客户端
- 是否集中发生在某个时间段

## 为什么客户端一直超时

建议优先检查以下项目：

- Base URL 是否正确
- 当前模型是否适合长上下文或交互式场景
- 当前分组是否偏向低成本
- 当前对话是否过长

对于开发工具类客户端，还需额外检查：

- 是否发生自动重试
- 是否默认附带了当前模型不支持的参数
- 是否缓存了旧配置

## 为什么控制台可以登录，但 API 不可用

控制台访问与 API 调用属于两条不同链路。

- 控制台可用，仅说明前台登录与页面访问正常
- API 可用，还要求令牌、Base URL、请求头与模型权限均正确

常见误区是将控制台可用直接等同于 API 可用。  
API 是否正常，应通过最小请求单独验证。

## 为什么 `401` 和 `403` 看起来很像

两者都表现为权限相关问题，但含义不同：

- `401` 更接近“身份未正确建立”
- `403` 更接近“身份已识别，但当前行为不被允许”

因此：

- `401` 先检查 token 与请求头
- `403` 先检查模型权限、分组与来源限制

## 为什么 `429` 一直出现

常见原因包括：

- 同一客户端短时间内请求过快
- 多个客户端共用同一令牌
- 当前分组更适合低成本，而非高并发
- 上游本身正在限流

更有效的处理方式通常不是简单增加重试次数，而是：

- 使用指数退避
- 为不同客户端拆分令牌
- 切换更适合当前场景的分组

## 为什么更换新令牌后仍然无法使用

该现象通常说明问题未必出在令牌本身。

还应继续检查：

- Base URL 是否正确
- 客户端是否仍缓存旧配置
- 当前模型是否仍填写错误
- 当前客户端接法是否正确

## 哪些报错适合重试

一般而言：

- 适合重试：`429`、`502`、`503`、`504`
- 不适合重试：`400`、`401`、`403`、`404`

如果已经确认是参数错误或权限不足，继续重试只会增加时间与额度成本。

## 遇到报错时的最短动作

1. 先查看状态码
2. 先执行 `GET /models`
3. 再发送一条最小 `chat/completions`
4. 最后回到具体客户端继续排查
