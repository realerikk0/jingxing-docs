# 令牌管理

令牌是井陉最核心的日常管理对象之一。对外，令牌决定谁可以访问 API；对内，令牌决定请求落在哪个分组、消耗哪部分额度，并形成怎样的审计记录。  
默认建议为“一个用途一个令牌”，而非“一个人一个令牌”。

## 管理目标

合格的令牌管理应满足以下要求：

- 名称能够直接标识用途
- 分组与使用场景一致
- 额度与有效期边界清晰
- 模型权限仅开放必要范围
- 泄露后可以快速停用与替换
- 日志中能够明确识别来源、用途与调用行为

## 创建令牌前需要确认的 6 项信息

### 1. 名称

建议直接采用“用途 + 环境 + 客户端”的命名方式。

例如：

- `cursor-dev`
- `backend-prod`
- `batch-embedding`
- `ops-claude-gateway`

该命名方式有助于在调用日志、额度消耗和异常流量排查中快速定位。

### 2. 分组

分组决定令牌最终可见的模型范围、价格策略以及稳定性目标。  
如尚未完成分组规划，建议先阅读[模型分组](/docs/platform/model-groups)。

推荐做法如下：

- 本地开发令牌使用开发分组
- 线上服务令牌使用生产分组
- 高成本模型令牌使用专用分组

不建议将不同风险等级或不同成本等级的流量混用在同一个令牌中。

### 3. 额度

首次测试阶段，额度仅需满足最小验证需求；进入正式使用后，应按用途设置不同边界：

- 临时测试令牌：小额度、短有效期
- 本地长期开发令牌：中等额度、定期审查
- 生产服务令牌：独立额度、单独监控

如平台支持“无限额度”令牌，也仅建议向少数明确用途开放，不建议作为默认配置。

### 4. 过期时间

过期时间并非可选装饰项，而是最有效的风险收口手段之一。

推荐做法：

- 本地调试：几天至几周
- 团队级工具：按月或按季度轮换
- CI / 自动任务：配合固定轮换计划

### 5. 模型限制

如某个令牌仅需访问少量模型，则应仅开放相应模型。该做法至少具有以下收益：

- 降低误用高成本模型的概率
- 降低客户端误选模型的概率
- 缩小问题排查范围

### 6. IP 或来源限制

如平台已启用来源限制能力，建议优先对以下类型令牌启用：

- 线上服务
- CI / 自动任务
- 固定出口服务器

个人本地开发令牌通常不宜设置过严的 IP 绑定，以免在切换网络或办公地点时造成误伤。

## 推荐的令牌拆分方式

### 按环境拆分

- `dev`
- `staging`
- `prod`

### 按客户端拆分

- `cursor`
- `cherry-studio`
- `backend`
- `batch-job`

### 按风险等级拆分

- 低成本通用令牌
- 高额度生产令牌
- 高成本模型专用令牌

如果一个令牌同时承担个人开发、团队共享、线上服务和高成本模型访问等多种职责，后续维护复杂度通常会明显上升。

## 轮换与停用

### 什么时候应该轮换

- 令牌被发到了不应到达的位置
- 团队成员离开项目
- 客户端配置曾被共享
- 怀疑令牌已经泄露
- 安全策略要求定期轮换

### 正确的轮换顺序

1. 先创建新令牌
2. 在客户端或服务端完成替换
3. 使用 `GET /models` 或最小请求验证新令牌
4. 观察一段时间，确认旧流量已经消失
5. 最后停用旧令牌

不建议先删除旧令牌，再反向排查仍在使用的系统；该做法容易直接中断线上流量。

### 遇到异常时怎么处理

如观察到异常高消耗、陌生来源或非预期模型调用，建议按以下顺序处理：

1. 先停用可疑令牌
2. 再检查日志与来源
3. 再决定是否调整分组、模型限制或 IP 限制

## 常见问题

### 为什么新令牌无法调用

建议优先检查：

- 令牌是否处于启用状态
- 请求头是否采用 `Authorization: Bearer ...`
- 当前分组是否允许目标模型
- 令牌是否设置了过窄的模型限制
- 令牌是否已经过期或额度不足

### 为什么同一个令牌在不同客户端表现不同

通常并非同一令牌失效，而是不同客户端的请求方式存在差异，例如：

- 某些客户端会先拉取模型列表
- 某些客户端会额外附带流式参数
- 某些客户端会默认切换到其他模型
- 某些客户端会缓存旧配置

因此，建议为不同客户端单独签发令牌。

### 为什么调用后余额没有立即变化

账单结算、缓存命中、流式结束时点和失败请求处理均可能影响展示时机。  
如短时间内余额未变化，应结合调用日志与账单页一并判断。

## 推荐操作顺序

1. 先确定分组
2. 再创建新令牌
3. 配置额度、过期时间与模型限制
4. 使用 `GET /models` 验证基础连通性
5. 再接入客户端或服务
6. 最后观察日志与消耗
