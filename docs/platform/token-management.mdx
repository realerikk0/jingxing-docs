# 令牌管理

令牌是井陉里最重要的日常管理对象之一。对外，它决定谁可以调用 API；对内，它决定调用会落在哪个分组、消耗哪部分额度、留下怎样的审计记录。  
一条简单但很重要的原则是：**不要把“一个人一个令牌”当成默认做法，而要把“一个用途一个令牌”当成默认做法。**

## 一个好令牌应该具备什么

- 名称能一眼看出用途
- 分组与使用场景一致
- 额度和有效期有明确边界
- 模型权限只开放必要范围
- 泄露后可以快速停用与替换
- 能在日志里清楚看出是谁、从哪、在做什么

## 创建令牌时要确认的 6 件事

### 1. 名称

建议把名称直接写成“用途 + 环境 + 客户端”。

例如：

- `cursor-dev`
- `backend-prod`
- `batch-embedding`
- `ops-claude-gateway`

这样后面看调用日志、配额消耗和异常流量时，不需要额外翻备注就能定位。

### 2. 分组

分组决定令牌最终能看到哪些模型、价格怎么计算、是否更偏向低成本还是高稳定性。  
如果你还没看过分组文档，建议先回到[模型分组](/docs/platform/model-groups)。

一个很实用的经验是：

- 本地开发令牌，用开发分组
- 线上服务令牌，用生产分组
- 高成本模型令牌，单独放专用分组

不要把不同风险等级、不同成本等级的流量混在同一个令牌里。

### 3. 额度

首次测试时，额度只要足够跑通即可；进入正式使用后，再按用途设置不同边界：

- 临时测试令牌：小额度、短有效期
- 本地长期开发令牌：中等额度、按月审查
- 生产服务令牌：独立额度、单独监控

如果你的平台使用“无限额度”令牌，也建议仅对少数明确用途开放，不要把它当默认配置。

### 4. 过期时间

过期时间不是可有可无的装饰字段，它是最省心的风险收口方式之一。

推荐做法：

- 本地调试：几天到几周
- 团队级工具：按月或按季度轮换
- CI / 自动任务：配合固定轮换计划

### 5. 模型限制

如果某个令牌只需要访问少数模型，就应该只开放少数模型。这样做至少有三个好处：

- 降低误用高成本模型的概率
- 降低客户端误选模型的概率
- 出问题时排查范围更小

### 6. IP 或来源限制

如果平台启用了来源限制能力，建议优先给以下类型的令牌开启：

- 线上服务
- CI / 自动任务
- 固定出口服务器

而个人本地开发令牌一般不适合做过严的 IP 绑定，否则切网络、切办公地点后很容易误伤自己。

## 推荐的令牌拆分方式

### 按环境拆

- `dev`
- `staging`
- `prod`

### 按客户端拆

- `cursor`
- `cherry-studio`
- `backend`
- `batch-job`

### 按风险等级拆

- 低成本通用令牌
- 高额度生产令牌
- 高成本模型专用令牌

如果一个令牌同时承担“个人开发 + 团队共享 + 线上服务 + 高成本模型”四种用途，那它几乎一定会在后续运维中制造麻烦。

## 轮换与停用

### 什么时候应该轮换

- 令牌被发到不该去的地方
- 团队成员离开项目
- 客户端配置曾被共享
- 你怀疑它可能已经泄露
- 例行安全策略要求固定周期轮换

### 正确的轮换顺序

1. 先创建新令牌
2. 在客户端或服务端完成替换
3. 用 `GET /models` 或一条最小请求验证新令牌
4. 观察一段时间确认没有旧流量继续打进来
5. 最后再停用旧令牌

不要先删旧令牌再去找哪里还在用它，这种做法最容易把线上流量一起打断。

### 遇到异常时怎么处理

如果你看到异常高消耗、陌生来源、非预期模型调用，优先顺序通常是：

1. 先停用可疑令牌
2. 再看日志和来源
3. 再决定是否要调整分组、模型限制或 IP 限制

## 常见问题

### 为什么新令牌无法调用

优先检查：

- 令牌是否真的处于启用状态
- 请求头是否用了 `Authorization: Bearer ...`
- 当前分组是否允许目标模型
- 令牌是否设置了过窄的模型限制
- 令牌是否已经过期或额度不足

### 为什么同一个令牌在不同客户端表现不同

通常不是“同一个令牌忽然失效”，而是不同客户端的请求方式不一样，例如：

- 有的客户端会先拉模型列表
- 有的客户端会额外带流式参数
- 有的客户端会默认切到另一个模型
- 有的客户端会缓存旧配置

这也是为什么建议给不同客户端单独发令牌。

### 为什么调用后余额没有立刻变化

账单、缓存命中、流式结束结算、失败请求处理，都可能影响展示时机。  
如果短时间内看起来“不动”，不要立刻判断成未扣费，先结合调用日志和账单页一起看。

## 推荐操作顺序

1. 先确定分组
2. 再创建新令牌
3. 再设置额度、过期时间和模型限制
4. 先用 `GET /models` 验证
5. 再接入客户端或服务
6. 最后观察日志和消耗
