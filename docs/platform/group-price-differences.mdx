# 分组的特殊性及价格差异

很多用户第一次接井陉时，最容易困惑的就是这件事：

**为什么同一个模型，在不同分组里价格不一样、速度不一样、甚至可用性都不一样？**

答案很直接：分组不是标签，而是策略。

## 先说结论

- 分组不同，价格可以不同
- 分组不同，模型范围可以不同
- 分组不同，稳定性和速度目标也可以不同
- 所以“同模型同价格”从来都不该被默认假设

## 分组的“特殊性”到底是什么

你可以把分组理解成“面向某种任务的整套运营配置”，它通常同时决定：

- 哪些模型可以调用
- 每个模型按什么倍率计费
- 是更偏向低成本，还是更偏向高稳定
- 是否适合桌面客户端、后端服务、批处理或高推理任务

也就是说，分组的特殊性不是表面名字，而是背后的业务目标。

## 为什么同一个模型会在不同分组价格不同

最常见的原因有 4 类：

### 1. 适用场景不同

一个分组可能是给本地开发用的，一个分组可能是给生产服务用的。  
前者更重视成本，后者更重视可用性和稳定性。

### 2. 稳定性目标不同

某些分组可能更强调：

- 低失败率
- 更长时间稳定输出
- 更适合交互式客户端

这类分组的价格通常不会和纯低成本线路一样。

### 3. 模型权限不同

有些分组会开放高成本模型，有些不会。  
一旦分组本身就是“高性能 / 高成本模型专用”，整体价格感受自然会完全不同。

### 4. 客户端行为不同

即使模型名相同，如果某个客户端更容易：

- 自动重试
- 拉更长上下文
- 频繁切模型

你也会主观感受到“这个分组怎么更贵”，但问题其实不只在模型上。

## 应该怎么看价格差异

不要只看“数字高低”，更应该一起看这三件事：

| 你要看什么 | 为什么重要 |
| --- | --- |
| 分组适合什么任务 | 决定它是不是该给你当前场景用 |
| 当前模型的倍率 | 决定单次调用的大概成本 |
| 客户端行为 | 决定最终真实消耗是不是被放大了 |

## 最常见的三种误判

### 误判 1：价格高就是不划算

不一定。  
如果某个分组显著更稳定、更适合交互式工具或更适合生产，那它价格高未必是不划算，反而可能更省排障成本。

### 误判 2：价格低就适合所有场景

也不对。  
低成本分组常常更适合：

- 本地测试
- 批量任务
- 非关键流量

不一定适合长对话、长输出、工具调用和团队高依赖场景。

### 误判 3：只要模型名一样，结果就应该一样

结果和体验除了模型，还会受到：

- 分组策略
- 参数
- 客户端
- 上下文长度

的共同影响。

## 推荐的选择方式

### 如果你是开发测试

优先选：

- 默认通用分组
- 低成本测试分组

### 如果你是桌面聊天客户端

优先选：

- 通用聊天分组
- 高稳定聊天分组

### 如果你是编码助手

优先选：

- 编码助手专用分组
- 高稳定聊天分组

### 如果你是后端生产服务

优先选：

- 生产分组
- 高稳定分组

## 推荐做法

最稳的方式不是“所有人都用一个便宜分组”，而是：

1. 测试场景一个分组
2. 桌面客户端一个分组
3. 生产服务一个分组
4. 高成本模型单独一个分组

这样你后面看账单、看日志、看失败率，才有意义。

## 用户实际操作时应该怎么判断

如果你现在已经站在控制台里，不知道选哪个分组，按这个顺序看：

1. 先看你当前用的是哪个客户端
2. 再看你是测试还是生产
3. 再看是否需要高成本模型
4. 最后才比较价格
