# 如何新建指定分组的令牌

本文用于说明：在已经明确目标分组的前提下，如何创建一个可用于指定客户端或服务的独立令牌，并完成基础验证。

如尚未确定应使用哪个分组，建议先阅读：

- [分组详细表格](/docs/platform/group-table)
- [分组的特殊性及价格差异](/docs/platform/group-price-differences)

## 前置条件

创建令牌前，建议先明确以下信息：

- 令牌的使用对象
- 目标分组
- 令牌对应的客户端或服务类型

在用途尚未明确的情况下直接创建大量令牌，通常会增加后续审计、排障和额度管理的复杂度。

## 命名建议

建议采用以下命名规则：

`用途 + 环境 + 客户端`

示例：

- `cursor-dev`
- `cherry-desktop`
- `backend-prod`
- `openwebui-team`

该命名方式便于后续在日志、账单和权限管理中快速识别令牌用途。

## 创建步骤

### 1. 进入令牌管理页面

登录控制台后，进入令牌管理页面。若当前账号没有创建权限，应先确认权限配置。

### 2. 新建令牌

建议按照以下顺序填写主要字段：

1. 名称
2. 分组
3. 额度
4. 过期时间
5. 模型限制
6. 其他附加限制

### 3. 明确选择分组

分组是创建过程中的关键字段。应确认当前选中的分组与实际用途一致，而不是沿用上一次使用的默认值。

常见对应关系如下：

- Cursor：编码或高稳定聊天分组
- Cherry Studio / Lobe Chat / Open WebUI：通用聊天分组
- 后端生产服务：生产分组
- 离线批处理：低成本或批处理分组

### 4. 设置额度

建议根据用途设定额度：

- 本地测试：小额度
- 日常客户端：中等额度
- 生产服务：独立额度并单独观察

若尚不明确是否需要无限额度，建议暂不启用。

### 5. 设置过期时间

短期测试令牌建议设置较短有效期；长期使用令牌可根据环境要求设置较长周期。

### 6. 配置模型限制

如该令牌仅服务于特定客户端，建议仅开放该客户端需要使用的模型。例如：

- Cursor：少量编码模型
- Cherry Studio：少量聊天模型
- Open WebUI：先开放聊天模型，后续再扩展

### 7. 保存并妥善存放令牌

创建完成后，应立即将令牌保存至安全位置。推荐的保存方式包括：

- 本地环境变量
- 密钥管理工具
- 服务端环境变量

不建议将令牌直接保存在聊天记录、临时文档或代码仓库中。

## 创建后的验证步骤

### 验证 1：检查模型列表

```bash
curl https://api.jingxing.uk/v1/models \
  -H "Authorization: Bearer <YOUR_TOKEN>"
```

应重点确认：

- 返回模型是否符合目标分组预期
- 是否出现不应开放的模型
- 是否缺少业务实际需要的模型

### 验证 2：发送最小请求

```bash
curl https://api.jingxing.uk/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <YOUR_TOKEN>" \
  -d '{
    "model": "gpt-5.4",
    "messages": [{"role": "user", "content": "你好"}]
  }'
```

该步骤用于确认：

- 令牌本身可用
- 分组与模型权限匹配
- 接入链路已具备最基本可用性

## 常见问题

### 分组选择错误

这是创建令牌时最常见的问题之一。建议在保存前再次核对分组与用途是否一致。

### 客户端仍在使用旧令牌

在桌面客户端或长期运行服务中较为常见，应确认目标系统已更新为新令牌。

### 模型限制过窄

若创建后 `GET /v1/models` 返回结果明显异常，应优先检查模型限制设置。

### 额度设置过低

过低额度可能导致验证阶段即出现失败，从而误判为分组或模型问题。

## 实施建议

建议按以下顺序执行：

1. 明确用途
2. 明确分组
3. 创建令牌
4. 验证模型列表
5. 发送最小请求
6. 最后接入客户端或服务
