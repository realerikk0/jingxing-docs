# 错误与重试

当你看到报错时，先不要急着把问题归因到“令牌坏了”或“平台挂了”。  
井陉场景里的问题，通常可以拆成 4 层：

1. 请求有没有正确到达平台
2. 请求参数是不是合法
3. 当前令牌、分组、模型权限是否匹配
4. 上游渠道是否当时不可用、超时或限流

## 先记住这个重试原则

**不是所有错误都适合自动重试。**

- `400`、`401`、`403`、`404` 这类明显配置或参数问题，通常不该重试
- `429`、`502`、`503`、`504` 这类更偏容量或上游波动的问题，通常才适合有限次重试
- `500` 要先区分是平台内部错误，还是上游返回的通用失败

## 常见状态码怎么处理

| 状态码 | 常见含义 | 适不适合自动重试 | 优先动作 |
| --- | --- | --- | --- |
| 400 | 参数错误、模型名错误、请求体不合法 | 不建议 | 先改请求 |
| 401 | 令牌无效、缺少鉴权 | 不建议 | 先查令牌和请求头 |
| 403 | 令牌被禁用、权限不足、来源受限 | 不建议 | 先查权限和策略 |
| 404 | 地址写错、路径不对、模型不可见 | 不建议 | 先查 URL 和模型 |
| 429 | 频率限制、额度限制、上游限流 | 建议有限重试 | 降速、退避、换分组 |
| 500 | 平台或上游内部失败 | 视情况而定 | 查日志、缩小请求 |
| 502 | 网关 / 上游异常 | 建议有限重试 | 退避重试或切模型 |
| 503 | 服务暂不可用 | 建议有限重试 | 等待恢复或切分组 |
| 504 | 上游超时 | 建议有限重试 | 缩短上下文、换更稳分组 |

## 按问题类型排查

### 401 未认证

最常见原因：

- 忘了带 `Authorization: Bearer ...`
- 令牌复制不完整
- 令牌已经过期
- 令牌已经停用

排查顺序：

1. 先看请求头
2. 再看当前令牌状态
3. 再用 `GET /models` 做最小验证

### 403 无权限

最常见原因：

- 当前令牌没有这个模型权限
- 当前令牌所在分组不允许该模型
- 开了来源限制但当前来源不在允许范围

这类问题一般不应该重试，因为重试不会改变权限本身。

### 400 参数错误

最常见原因：

- `model` 填错
- `messages` 结构不合法
- JSON 非法
- 客户端默认加了当前模型不支持的参数

如果你是通过第三方客户端接入，尤其要留意“客户端默认参数”和“当前模型实际支持范围”之间的差异。

### 404 地址或模型不对

最常见错误是把：

- `https://api.jingxing.uk`

当成完整地址，而漏掉了：

- `https://api.jingxing.uk/v1`

也有一类情况是模型名在当前令牌下不可见，客户端把它解释成“找不到”。

### 429 频率限制或额度限制

这是最适合做自动重试的一类，但要有边界。

推荐做法：

- 使用指数退避
- 控制最大重试次数
- 避免多个客户端一起无脑重试

如果你已经确定是当前分组承压或当前模型承压，直接换更稳的分组或模型，通常比反复重试更有效。

### 500 / 502 / 503 / 504 上游或链路问题

这类问题更像“链路异常”，而不是“配置错了”。

优先顺序通常是：

1. 先缩小请求体
2. 再减少上下文长度
3. 再检查是否只在某个模型或某个分组下出现
4. 最后再决定是否要自动重试或切换模型

## 推荐的自动重试策略

### 适合重试的场景

- `429`
- `502`
- `503`
- `504`
- 少量可恢复的 `500`

### 不适合重试的场景

- `400`
- `401`
- `403`
- `404`
- 明显的模型权限或参数错误

### 建议的重试方式

1. 第一次失败后短暂等待
2. 第二次失败后增加退避时间
3. 达到上限后停止，并记录日志

如果你的客户端支持自动重试，最好同时记录：

- 原始状态码
- 重试次数
- 最终是否成功
- 用的是哪个模型和分组

否则后面会很难解释“为什么明明只发了一次请求，却消耗了多次”。

## 遇到报错时最短排查路径

1. 先看状态码
2. 再看是不是配置问题还是链路问题
3. 用 `GET /models` 验证令牌和基础连通性
4. 再发一条最小 `chat/completions` 请求
5. 确认是否只在某个模型、某个客户端、某个分组下复现

## 客户端侧建议

### 对开发工具类客户端

像 Cursor、Cherry Studio、Open WebUI 这类客户端，最容易遇到的问题不是“平台完全不可用”，而是：

- 缓存旧模型列表
- 默认切到不可用模型
- 自动附带当前模型不支持的参数
- 在网络波动时偷偷重试

所以排查时一定要同时看：

- 客户端当前选中的模型
- Base URL 是否正确
- API Key 是否真的是当前令牌
- 客户端是否存在自动重试或模型缓存
