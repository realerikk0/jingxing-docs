# 错误与重试

井陉场景中的错误通常可分为四个层级。排查时建议按层级逐步收缩范围，而不是直接将问题归因为令牌失效或平台不可用。

1. 请求是否正确到达平台
2. 请求参数是否合法
3. 令牌、分组与模型权限是否匹配
4. 上游渠道是否处于限流、超时或不可用状态

## 重试原则

**并非所有错误都适合自动重试。**

- `400`、`401`、`403`、`404` 等明显的配置或参数问题，通常不应自动重试
- `429`、`502`、`503`、`504` 等更偏向容量波动或上游抖动的问题，通常适合有限次重试
- `500` 应先区分是平台内部错误，还是上游返回的通用失败

## 常见状态码处理建议

| 状态码 | 常见含义 | 是否建议自动重试 | 优先动作 |
| --- | --- | --- | --- |
| 400 | 参数错误、模型名错误、请求体不合法 | 不建议 | 先修正请求 |
| 401 | 令牌无效、缺少鉴权 | 不建议 | 先检查令牌和请求头 |
| 403 | 令牌被禁用、权限不足、来源受限 | 不建议 | 先检查权限与策略 |
| 404 | 地址错误、路径错误、模型不可见 | 不建议 | 先检查 URL 与模型名 |
| 429 | 频率限制、额度限制、上游限流 | 建议有限重试 | 降速、退避、调整分组 |
| 500 | 平台或上游内部失败 | 视情况而定 | 查日志并缩小请求 |
| 502 | 网关或上游异常 | 建议有限重试 | 退避重试或切换模型 |
| 503 | 服务暂不可用 | 建议有限重试 | 等待恢复或切换分组 |
| 504 | 上游超时 | 建议有限重试 | 缩短上下文并切换更稳定的分组 |

## 按问题类型排查

### 401 未认证

常见原因包括：

- 未携带 `Authorization: Bearer ...`
- 令牌复制不完整
- 令牌已经过期
- 令牌已经停用

建议排查顺序：

1. 先检查请求头
2. 再确认令牌状态
3. 使用 `GET /models` 完成最小验证

### 403 无权限

常见原因包括：

- 当前令牌没有目标模型权限
- 当前分组未开放目标模型
- 已启用来源限制，但当前来源不在允许范围内

此类问题通常不应重试，因为重试不会改变权限配置本身。

### 400 参数错误

常见原因包括：

- `model` 填写错误
- `messages` 结构不合法
- JSON 非法
- 客户端默认附带了当前模型不支持的参数

如通过第三方客户端接入，应重点核对“客户端默认参数”与“当前模型实际支持范围”之间的差异。

### 404 地址或模型错误

常见错误之一是将：

- `https://api.jingxing.uk`

误认为完整地址，而遗漏：

- `https://api.jingxing.uk/v1`

另一类常见情况是模型名称在当前令牌下不可见，而客户端将其解释为“找不到”。

### 429 频率限制或额度限制

该类问题通常适合自动重试，但必须设置明确边界。

推荐做法：

- 使用指数退避
- 设置最大重试次数
- 避免多个客户端同时无边界重试

如已确认当前分组或当前模型处于承压状态，切换至更稳定的分组或模型通常比重复重试更有效。

### 500 / 502 / 503 / 504 上游或链路问题

该类问题更接近链路异常，而非参数配置错误。

建议优先执行以下操作：

1. 先缩小请求体
2. 再减少上下文长度
3. 再检查是否只在某个模型或某个分组下出现
4. 最后决定是否自动重试或切换模型

## 推荐的自动重试策略

### 适合重试的场景

- `429`
- `502`
- `503`
- `504`
- 少量可恢复的 `500`

### 不适合重试的场景

- `400`
- `401`
- `403`
- `404`
- 明显的模型权限或参数错误

### 建议的重试方式

1. 第一次失败后短暂等待
2. 第二次失败后增加退避时间
3. 达到上限后停止，并记录日志

如客户端支持自动重试，建议同时记录：

- 原始状态码
- 重试次数
- 最终是否成功
- 实际使用的模型与分组

否则后续很难准确解释请求次数与消耗之间的对应关系。

## 最短排查路径

1. 先看状态码
2. 判断问题属于配置类还是链路类
3. 使用 `GET /models` 验证令牌与基础连通性
4. 再发送一条最小 `chat/completions` 请求
5. 确认是否只在某个模型、某个客户端或某个分组下复现

## 客户端侧补充检查

### 对开发工具类客户端

对于 Cursor、Cherry Studio、Open WebUI 等客户端，更常见的问题并非平台整体不可用，而是：

- 缓存旧模型列表
- 默认切换到不可用模型
- 自动附带当前模型不支持的参数
- 在网络波动时发生自动重试

因此，排查时应同时检查：

- 客户端当前选中的模型
- Base URL 是否正确
- API Key 是否为当前令牌
- 客户端是否存在自动重试或模型缓存
