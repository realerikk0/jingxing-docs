# 分组详细表格

这一页不是教你理解抽象概念，而是帮你直接回答一个问题：**我现在这个场景，到底该选哪一类分组。**

先说最重要的原则：

- 不要先看分组名字，要先看你的用途
- 不要先问“哪个最便宜”，要先问“我要接哪个客户端、跑什么任务”
- 如果控制台显示的实际分组名称和本文不一样，以控制台为准

## 先按场景选，再去控制台对名字

| 你的目标 | 推荐分组特征 | 适合的客户端 / 场景 | 优先关注 | 不建议 |
| --- | --- | --- | --- | --- |
| 先把 API 跑通 | 默认分组、通用分组、基础分组 | `curl`、SDK、自建后端 | 模型可见、错误少 | 一上来就选高成本专用分组 |
| 日常聊天和桌面客户端 | 通用对话分组、聊天分组 | Cherry Studio、Lobe Chat、Open WebUI | 响应速度、模型稳定性 | 用批处理分组硬跑聊天 |
| 编码助手 | Cursor / Claude Code / Codex 专用分组，或高稳定聊天分组 | Cursor、Claude Code、OpenAI Codex | 长上下文、稳定性、工具调用 | 用最便宜但不稳的线路 |
| 本地开发测试 | 开发分组、低成本分组 | 本地脚本、联调、验证 | 成本、可用性 | 和生产共用分组 |
| 线上生产服务 | 生产分组、高稳定分组 | Web 服务、API 服务、机器人 | 稳定性、日志、审计 | 和个人调试共用 |
| 批量任务 / 离线任务 | 批处理分组、低成本分组 | 批量改写、Embedding、离线生成 | 成本、并发、重试策略 | 追求超低延迟 |
| 高成本模型专用 | 高性能 / 高等级 / 专用模型分组 | 高推理模型、图片、视频 | 成本边界、权限控制 | 开给所有普通客户端 |

## 常见客户端怎么选

| 客户端 | 第一推荐 | 第二推荐 | 备注 |
| --- | --- | --- | --- |
| Cursor | 编码助手专用分组 | 高稳定聊天分组 | 不建议和普通聊天共用令牌 |
| Claude Code | Claude / 编码专用分组 | 团队级网关分组 | 更适合团队统一代理接入 |
| Cherry Studio | 通用聊天分组 | 低成本聊天分组 | 适合桌面日常使用 |
| Lobe Chat | 通用聊天分组 | 生产聊天分组 | 自部署时建议独立令牌 |
| Open WebUI | 通用聊天分组 | 生产聊天分组 | 后续若接图像 / RAG 再细分 |
| 后端 API | 生产分组 | 默认通用分组 | 先稳定，再考虑省钱 |

## 同样是“聊天”，为什么还要分组

因为“聊天”不是一个单一场景。你至少会遇到这几种差异：

- 本地开发只想便宜
- 线上服务更想稳
- 编码助手更吃上下文和连续对话
- 图片 / 视频前端会混入更高成本模型

所以分组真正要表达的是：

- 适用任务
- 模型范围
- 稳定性目标
- 成本边界

## 第一次接入时怎么选

如果你完全不确定，建议用下面这个顺序：

1. 先选默认或最通用的分组
2. 先跑通 `GET /models`
3. 再发第一条 `chat/completions`
4. 再根据客户端类型换到更合适的专用分组

不要在第一步就试图找“最极致最优”的分组，那通常只会让你更慢跑通。

## 什么时候应该单独再建一个分组

出现下面情况时，就应该认真考虑拆分组了：

- 生产流量和测试流量混在一起
- 团队里有人经常误用高成本模型
- 同一个客户端既跑普通聊天又跑高推理任务
- 你已经很难从账单和日志判断到底是谁在消耗

## 最常见的误区

### 误区 1：只看名字

分组名称只是入口，不是全部。  
真正要看的是它背后的模型范围、价格和适用场景。

### 误区 2：所有客户端共用一个分组

这会让你后面很难判断：

- 谁在报错
- 谁在超时
- 谁在烧额度

### 误区 3：把最便宜当成最合适

低成本分组常常适合测试和批处理，不一定适合交互式聊天或编码助手。

## 推荐动作

如果你现在就要开始操作，按这个顺序最稳：

1. 先确认你要接的是哪类客户端
2. 从上表选择最接近的分组特征
3. 到控制台里找到对应分组
4. 用该分组单独新建令牌
5. 跑 `GET /models` 验证
