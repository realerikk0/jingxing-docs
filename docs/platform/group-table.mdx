# 分组详细表格

本页用于回答一个实际问题：在不同使用场景下，应当选择哪一类分组。分组的判断依据不应仅限于名称或单一价格因素，而应综合考虑客户端类型、任务类型、稳定性要求和成本边界。

## 选择原则

建议优先基于以下维度判断分组：

- 使用目标：聊天、编码、生产服务、批处理、图像、视频
- 客户端类型：桌面客户端、Web 前端、后端服务、命令行工具
- 关注重点：稳定性、成本、并发能力、可审计性

如控制台中的实际分组名称与本文示例不完全一致，应以控制台当前配置为准。

## 按场景选择分组

| 使用目标 | 推荐分组特征 | 适用客户端 / 场景 | 优先关注 | 不建议 |
| --- | --- | --- | --- | --- |
| API 初次验证 | 默认分组、通用分组、基础分组 | `curl`、SDK、自建后端 | 模型可见性、请求成功率 | 首轮即使用高成本专用分组 |
| 日常聊天和桌面客户端 | 通用聊天分组、对话分组 | Cherry Studio、Lobe Chat、Open WebUI | 响应速度、模型稳定性 | 使用批处理分组执行交互式聊天 |
| 编码助手 | 编码专用分组、高稳定聊天分组 | Cursor、Claude Code、Codex 类客户端 | 长上下文、稳定性、工具调用能力 | 使用低成本但不稳定的线路 |
| 本地开发测试 | 开发分组、低成本分组 | 本地脚本、联调、功能验证 | 成本、可用性 | 与生产服务共用分组 |
| 线上生产服务 | 生产分组、高稳定分组 | Web 服务、API 服务、机器人 | 稳定性、日志、审计能力 | 与个人调试流量共用 |
| 批量任务 / 离线任务 | 批处理分组、低成本分组 | 批量改写、Embedding、离线生成 | 成本、并发、重试策略 | 以低延迟为唯一目标 |
| 高成本模型专用 | 高性能分组、专用模型分组 | 高推理模型、图像、视频 | 成本边界、权限控制 | 面向所有普通客户端默认开放 |

## 按客户端选择分组

| 客户端 | 第一推荐 | 第二推荐 | 说明 |
| --- | --- | --- | --- |
| Cursor | 编码专用分组 | 高稳定聊天分组 | 建议单独令牌与独立分组 |
| Claude Code | Claude / 编码专用分组 | 团队级网关分组 | 适合通过企业代理统一接入 |
| Cherry Studio | 通用聊天分组 | 低成本聊天分组 | 适合日常桌面对话 |
| Lobe Chat | 通用聊天分组 | 生产聊天分组 | 自部署时建议单独令牌 |
| Open WebUI | 通用聊天分组 | 生产聊天分组 | 图像与 RAG 建议后续再拆分 |
| 后端 API 服务 | 生产分组 | 默认通用分组 | 优先稳定性，再优化成本 |

## 为什么相同“聊天”场景仍需要区分分组

聊天场景并非单一类别，其差异通常体现在：

- 本地测试侧重成本控制
- 线上服务侧重稳定性和可审计性
- 编码助手侧重长上下文与连续对话
- 图像与视频前端可能混入高成本模型

因此，分组的意义并非仅仅是命名，而是用于表达：

- 可用模型范围
- 成本边界
- 稳定性目标
- 适用客户端与任务类型

## 首次接入建议

若尚不明确具体分组，建议采用以下顺序：

1. 先选择默认或通用分组
2. 执行 `GET /v1/models`
3. 完成首条 `chat/completions` 验证
4. 再根据实际客户端与任务类型迁移到更适合的专用分组

## 何时需要新增独立分组

出现以下情况时，建议考虑单独拆分分组：

- 生产流量与测试流量混用
- 高成本模型容易被误调用
- 同一客户端同时承担普通聊天与高推理任务
- 账单与日志已难以区分流量来源

## 常见误区

### 仅根据名称选择分组

分组名称仅用于识别，真正应关注的是其背后的模型范围、价格和适用场景。

### 多个客户端共用同一分组

此做法会显著增加后续排障与审计成本，不利于定位额度消耗、超时和模型误用来源。

### 将最低价格视为唯一目标

低成本分组通常更适合测试和批处理任务，不一定适合编码助手或高频交互式聊天。

## 实施建议

建议按以下顺序实施：

1. 明确客户端类型与任务类型
2. 选择最接近的分组特征
3. 在控制台中确认实际分组名称
4. 使用该分组单独创建令牌
5. 通过 `GET /v1/models` 验证模型可见性
