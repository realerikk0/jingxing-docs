# 模型分组

分组是井陉最重要的管理概念之一。它不只是一个标签，而是一组能力、价格、模型范围和策略的组合。你可以把分组理解成“同一套调用规范下的不同服务档位”。

## 分组会影响什么

- 当前令牌能看到哪些模型
- 每个模型的价格或倍率
- 稳定性、延迟和可用性策略
- 团队内部如何做权限隔离和成本控制
- 某些高级能力是否开放，例如图片、视频或 Realtime

## 为什么需要分组

如果所有用户都共用同一套模型和同一套价格策略，平台很快就会遇到三个问题：

- 成本不可控
- 团队无法隔离
- 不同场景很难同时兼顾稳定性与价格

分组的作用，就是把这些差异管理前置。这样你可以：

- 给不同团队分配不同服务档位
- 给不同用途分配不同价格策略
- 在不改客户端代码结构的前提下切换可用模型范围

## 选择分组时要看什么

| 目标 | 关注点 | 推荐做法 |
| --- | --- | --- |
| 先跑通接入 | 可用模型范围、容错性 | 先选默认或最通用的分组 |
| 控制成本 | 单价、倍率、额度策略 | 单独为测试、批处理和研发建立低成本分组 |
| 保证稳定性 | 延迟、失败率、可用模型冗余 | 给生产流量使用更稳的分组 |
| 做团队隔离 | 令牌权限、审计、额度 | 每个团队或环境单独建分组和令牌 |

## 推荐实践

### 每种用途单独一组

最常见的做法不是“所有人共用一个大分组”，而是按用途拆开：

- 本地开发
- 生产服务
- 批处理或离线任务
- 高成本模型的专用调用

这样做的好处是：一旦某一类流量异常，你能更快知道是哪个场景出了问题。

### 每个分组配独立令牌

虽然一个用户可以持有多个令牌，但更推荐按分组或按用途分开发令牌。这样可以获得更清晰的审计与额度边界。

### 把“能不能用”与“适不适合用”分开

某个分组可见某个模型，不代表它就是当前场景最优解。文档、客户端和控制台最好同时写清楚：

- 当前分组有哪些模型可以调用
- 哪些模型适合交互式场景
- 哪些模型适合长输出或批量处理

## 如何创建指定分组的令牌

1. 先在控制台确认目标分组存在且你有使用权限
2. 新建令牌时明确选择该分组
3. 如有需要，再叠加模型限制或 IP 限制
4. 用 `GET /models` 验证当前令牌实际可见的模型范围

## 文档中的分组信息应如何维护

分组属于运营配置，天然会比代码变得更快。文档里建议这样处理：

- 原则、选择方法、使用边界写在正文里
- 具体可用分组、模型列表和价格表尽量来自控制台或自动化同步
- 如果正文和控制台显示不一致，以控制台展示为准
