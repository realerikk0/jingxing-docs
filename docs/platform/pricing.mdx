# 价格说明

井陉的价格体系本质上是“**模型成本 + 分组策略 + 平台计费规则**”的组合。  
这意味着：同样是一次调用，最终消耗并不只取决于模型名，还取决于你当时使用的是哪个分组、有没有命中缓存、是不是图片或视频能力，以及该请求是否真正被上游处理。

## 先看结论

- 控制台中的价格表始终比文档正文更准确
- 分组不同，价格可以不同
- 模型不同，倍率可以不同
- 不同能力类型，计费方式也可能不同
- 文档适合讲规则，控制台适合看最新数字

## 价格由什么决定

### 1. 模型本身

不同模型的输入、输出、缓存、图像生成、视频生成成本差异很大。  
这也是为什么“模型名接近”不代表“价格接近”。

### 2. 分组策略

同一模型在不同分组中，价格、倍率、可用性策略都可能不同。  
例如：

- 面向研发测试的分组，可能更强调低成本
- 面向线上生产的分组，可能更强调稳定性和冗余
- 面向高性能或高优先级场景的分组，可能价格更高

### 3. 能力类型

文本聊天、Embedding、图片、视频、音频/Realtime 的计费逻辑往往不是一套规则。

最常见的差异有：

- 文本模型按输入 / 输出计费
- Embedding 按输入量计费
- 图片按张数、尺寸或质量档位计费
- 视频按时长、分辨率或任务类型计费
- 音频 / Realtime 可能同时受时长和 token 影响

## 文档里应该怎么理解“额度”

如果你的平台前台展示的是额度而不是直接人民币，那么可以把额度理解成“平台内部统一记账单位”。

在实际使用中，你更需要关心的是三件事：

- 当前令牌还剩多少可用额度
- 这类请求大概会消耗多少额度
- 哪些场景最容易把额度消耗得比预期更快

## 为什么同一个模型有时感觉更贵

最常见的原因不是“系统乱扣费”，而是下面这些差异被忽略了：

- 这次走的是另一个分组
- 这次输出更长
- 这次没有命中缓存
- 这次调用的是图片、视频或音频能力
- 这次客户端自动重试了多次

## 最常见的计费误区

### 误区 1：只看模型名，不看分组

这是最容易产生误判的地方。  
如果两次请求不在同一分组，即使模型名一样，价格也不一定一样。

### 误区 2：只看单次成功响应，不看总调用链路

有些客户端在失败、超时、连接中断时会自动重试。  
如果你只盯着“最后一次成功响应”，会低估实际发生过的请求次数。

### 误区 3：把缓存命中当成必然

缓存命中通常会影响最终消耗，但它不是无条件发生的。  
提示词、上下文、参数和请求模式变化后，命中情况就可能改变。

## 建议怎么查看价格

### 看规则

规则看文档，重点理解：

- 分组会影响价格
- 能力类型会影响价格
- 客户端重试会影响总消耗

### 看最新数字

最新数字看控制台，重点查看：

- 当前分组价格
- 当前模型倍率
- 账单或消费记录
- 是否有特殊模型或特殊能力的单独说明

如果正文和控制台价格表不一致，以控制台为准。

## 建议的价格管理方式

### 开发环境

- 用单独的低成本分组
- 给本地调试单独令牌
- 尽量避免默认开放高成本模型

### 生产环境

- 独立令牌
- 独立分组
- 独立观察消耗趋势

### 高成本模型

- 单独建立高成本分组
- 单独发令牌
- 明确用途边界

## 常见问题

### 为什么失败了也感觉被扣了额度

要先区分失败发生在哪一层：

- 请求根本没到平台
- 平台收到但参数错误
- 平台已发给上游，上游处理中失败

不同阶段的失败，是否计费、如何展示，可能并不一样。  
排查时不要只看前端提示，最好同时看调用日志和账单记录。

### 为什么流式输出和非流式看起来不一样

流式请求的生命周期更长，客户端也更容易在网络波动时触发重试或中断。  
如果你在高并发场景下观察到消耗波动，先确认是不是流式行为本身带来的差异。

### 为什么图片、视频比文本贵很多

因为它们本来就不是同一类成本结构。  
不要用文本模型的单位成本去预估图片或视频请求。
