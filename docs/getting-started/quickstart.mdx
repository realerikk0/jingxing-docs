# 快速开始

本文用于完成井陉的首次接入验证，包括控制台登录、令牌创建、环境变量配置以及第一条 API 请求的发送。

## 前置条件

开始前，请确认已具备以下条件：

- 控制台地址：`https://jingxing.uk`
- API 地址：`https://api.jingxing.uk/v1`
- 一个可登录控制台的账号
- 创建 API 令牌所需的基础权限

## 第一步：登录控制台

登录控制台后，建议优先确认以下信息：

- 当前账号是否具备创建令牌的权限
- 当前账号默认可用的分组
- 当前余额或额度是否足以完成测试请求

## 第二步：创建令牌

进入令牌管理页面后，建议创建一个新的测试令牌，而不是直接复用历史令牌。该做法有助于：

- 分离首次验证与历史流量
- 降低额度、权限和分组排查的复杂度
- 为后续不同客户端建立独立审计边界

创建时建议至少确认以下字段：

- 令牌名称：如 `local-test`、`cursor-dev`、`ci-bot`
- 分组：选择符合当前验证目标的分组
- 额度：足以完成首轮验证
- 过期时间：临时测试可设置较短周期
- 模型限制：首次验证阶段不建议设置过窄

## 第三步：配置环境变量

建议将令牌保存为环境变量，而非直接写入代码仓库或脚本文件。

```bash
export JINGXING_API_KEY="<YOUR_TOKEN>"
export JINGXING_BASE_URL="https://api.jingxing.uk/v1"
```

该方式便于在 `curl`、脚本和第三方客户端之间复用同一组配置。

## 第四步：验证模型列表

建议优先执行模型列表查询，以确认鉴权和基础连通性已经建立。

```bash
curl "$JINGXING_BASE_URL/models" \
  -H "Authorization: Bearer $JINGXING_API_KEY"
```

如成功返回模型列表，通常说明以下条件已经满足：

- 域名可达
- 令牌有效
- 请求头正确
- 当前令牌具备基础调用权限

## 第五步：发送第一条聊天请求

```bash
curl "$JINGXING_BASE_URL/chat/completions" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $JINGXING_API_KEY" \
  -d '{
    "model": "gpt-5.4",
    "messages": [
      {"role": "system", "content": "你是一个简洁的助手。"},
      {"role": "user", "content": "请用一句话介绍井陉。"}
    ]
  }'
```

如返回结果中包含 `choices`，则说明首次 API 请求已成功建立。

## 常见问题

### 401 未认证

建议优先检查：

- 是否携带 `Authorization: Bearer ...`
- 令牌是否完整复制
- 当前令牌是否已停用、过期或无效

### 模型不存在或不可用

常见原因包括：

- 当前分组未开放该模型
- 令牌设置了模型限制
- 模型名称填写错误

### 429 或超时

建议检查：

- 当前分组是否适合高并发或长输出场景
- 当前模型是否适配当前分组
- 是否需要切换到更稳定或更高配额的分组
