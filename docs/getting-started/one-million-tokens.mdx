# 1M Token 是多少

在 AI API 中，`token` 是模型处理文本的基本计量单位。  
输入给模型的提示词、上下文、历史消息、代码、文档，以及模型返回的答案，都会被拆成 token 并计入用量。

`1M Token` 指的是 `1,000,000` 个 token，也就是一百万 token。它不是固定的一百万个汉字，也不是一百万个英文单词，而是一种由模型 tokenizer 计算出来的文本单位。

## 什么是 token

可以把 token 理解成“模型眼里的文字颗粒”：

- 一个英文单词可能是 1 个 token，也可能被拆成多个 token。
- 一个中文句子通常会被拆成多个 token，具体数量取决于模型的 tokenizer。
- 标点、空格、换行、代码符号也可能单独计入 token。
- 请求里的历史对话、系统提示词、工具调用参数、模型输出内容，都会消耗 token。

一个简单例子：

```txt
Hello, world!
```

这句话在人眼看来只有两个英文单词，但模型会把它拆成若干 token，例如单词、标点和空格片段。

再看一个中文例子：

```txt
请帮我总结这篇文章。
```

这句话不会按“词语”简单计数，而是会被 tokenizer 拆成模型可处理的片段。

## 粗略换算

不同模型、语言和内容类型的 token 计算方式会有差异。下面是常见的粗略估算：

| 内容类型 | 粗略关系 |
| --- | --- |
| 英文文本 | `1 token` 大约相当于 `4` 个英文字符 |
| 英文单词 | `100 tokens` 大约相当于 `75` 个英文单词 |
| 中文文本 | 通常比英文更难用“单词”估算，建议按实际用量记录判断 |
| 代码内容 | 符号、缩进、变量名都会计入 token，消耗可能高于普通短文本 |

因此，`1M Token` 可以粗略理解为：

- 约 `4,000,000` 个英文字符
- 约 `750,000` 个英文单词
- 大量中文文章、对话、代码和文档的累计处理量

## 一百万 token 相当于什么

以下例子用于建立直观概念，实际消耗会因语言、模型、提示词长度和输出长度而变化。

| 场景 | 单次大致消耗 | 1M Token 大约可支持 |
| --- | ---: | ---: |
| 很短的问答 | `100 - 300 tokens` | `3,000 - 10,000` 次 |
| 普通聊天对话 | `500 - 1,500 tokens` | `600 - 2,000` 次 |
| 较完整的代码解释 | `2,000 - 5,000 tokens` | `200 - 500` 次 |
| 一篇中长文章总结 | `3,000 - 8,000 tokens` | `125 - 330` 篇 |
| 一份产品文档问答 | `10,000 - 30,000 tokens` | `30 - 100` 份 |
| 一次较长会议纪要整理 | `15,000 - 40,000 tokens` | `25 - 65` 次 |
| 一个小型代码模块分析 | `20,000 - 80,000 tokens` | `12 - 50` 次 |

## 用文字量理解 1M Token

如果主要处理英文内容，`1M Token` 约等于 `75` 万个英文单词。这个量级已经接近多本书的总文字量。

例如：

- 一本 `80,000` 英文单词的普通书，大约需要 `100,000+ tokens` 处理。
- `1M Token` 大约可以覆盖数本普通英文书的文本量。
- 一篇 `1,000` 英文单词的文章，大约需要 `1,300+ tokens`。
- `1M Token` 大约可以覆盖数百篇中等长度英文文章。

如果主要处理中文内容，不建议直接用“词数”换算。更实用的理解方式是：

- 一篇几千字中文文章，通常会消耗数千 token。
- 一份几十页中文资料，通常会消耗数万 token。
- 一百万 token 足够完成大量文章总结、资料问答、客服回复、代码解释和日常辅助写作。

## 用对话次数理解 1M Token

一次 AI 对话的 token 消耗通常包括：

- 系统提示词
- 用户输入
- 历史对话上下文
- 模型输出

如果每次只是短问短答，例如：

```txt
用户：帮我写一段商品介绍。
模型：返回 100-200 字结果。
```

这类请求可能只消耗几百 token，`1M Token` 可以支持数千次。

如果每次都带上较长上下文，例如：

```txt
用户：请阅读下面 20 页资料，并生成摘要、风险点和待办事项。
模型：返回结构化总结。
```

这类请求可能一次就消耗几万 token，`1M Token` 可能只能支持几十次。

## 用开发场景理解 1M Token

对开发者来说，token 的价值不只在“能聊多少句”，更在于能处理多少上下文。

常见开发场景包括：

- 让模型解释一个函数、一个文件或一个模块。
- 让模型根据接口文档生成调用示例。
- 让模型阅读错误日志并给出排查路径。
- 让模型根据已有代码生成测试用例。
- 让模型对比两版代码差异并总结风险。

如果每次只处理一个小文件，消耗可能只有几千 token。  
如果一次把多个文件、日志、接口定义和需求说明都发给模型，消耗可能很快达到几万甚至十几万 token。

## 为什么 1M Token 有价值

一百万 token 的价值主要体现在三个方面：

- **接入门槛低**：足够覆盖大量短问答、日常写作、代码解释和资料总结。
- **上下文空间大**：可以处理更长的文档、更多历史对话和更复杂的开发上下文。
- **成本更可控**：以 token 为单位计量后，可以按模型、分组、令牌和实际调用量精细管理消耗。

对个人用户来说，`1M Token` 通常已经足够完成大量日常 AI 使用。  
对团队用户来说，`1M Token` 更适合用来评估项目、环境或客户端的真实消耗。

## 需要注意什么

- 输入和输出都会消耗 token。
- 模型输出越长，消耗越高。
- 历史上下文越长，每次请求可能越贵。
- 不同模型的 tokenizer 和价格可能不同。
- 同样是 `1M Token`，输入 token 和输出 token 的计费价格可能不同。

因此，查看用量时不要只看请求次数，还要看每次请求携带了多少上下文、返回了多少内容，以及调用的是哪个模型分组。
