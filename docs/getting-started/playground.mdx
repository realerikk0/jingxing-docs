# AI 工作台

AI 工作台适合做一件很实际的事：**不写代码，直接在网页里确认模型、分组和效果是不是符合预期。**

如果你是第一次接井陉，或者你已经拿到了令牌但还不想立刻写脚本，AI 工作台通常是最快的体验入口。

## AI 工作台适合做什么

- 用网页直接测试模型
- 快速比较不同模型或不同分组的输出差异
- 在不接 SDK 的情况下先验证“这个模型能不能用”
- 给运营、产品或非开发同学一个可直接操作的体验入口

## 它和 API 的区别

- AI 工作台更适合人工测试
- API 更适合程序、脚本、客户端和自动化服务

一句话理解：

- 你想“先体验”，用 AI 工作台
- 你想“正式接入”，用 API

## 什么时候应该先用 AI 工作台

以下几种情况，优先打开工作台会更省时间：

- 你不确定某个分组适不适合当前任务
- 你想先比较两个模型的表现
- 你想先确认平台侧不是完全不可用
- 你要给非开发同学演示

## 最短操作路径

### 第 1 步：登录控制台

先进入控制台，确认当前账号可正常访问用户工作区。

### 第 2 步：进入 AI 工作台

找到 AI 工作台入口后，先不要急着做复杂测试，第一轮只做最简单验证。

### 第 3 步：选择模型或分组

如果当前工作台允许你切换模型或分组，建议第一轮：

- 只选一个你已经知道可用的模型
- 不要同时混很多模型和很多参数

### 第 4 步：发一个最小问题

例如：

- “用一句话介绍井陉”
- “把这段文字总结成三点”
- “给这段内容起一个标题”

第一轮的目标不是测极限，而是先确认：

- 模型有响应
- 速度可接受
- 输出风格符合预期

## 推荐怎么用它做对比

### 对比模型

如果你想比较两个模型，不要一次改太多变量。  
最稳的方式是：

1. 提示词保持一致
2. 每次只换一个模型
3. 观察速度、长度、风格和稳定性

### 对比分组

如果工作台当前支持切分组，那更适合拿来判断：

- 哪个分组更稳
- 哪个分组更适合某类任务
- 哪个分组的速度和成本更符合当前目标

## 常见误区

### 误区 1：工作台能用，就代表 API 一定没问题

不一定。  
工作台和 API 虽然共用同一个平台，但实际排障时还是两条路径。

### 误区 2：工作台慢，就说明模型一定差

不一定。  
你要先分清楚：

- 是当前模型本来就慢
- 还是当前分组更偏低成本
- 还是你的问题本来就很长

### 误区 3：在工作台里测一个模型，就代表所有客户端都一样

也不一定。  
像 Cursor、Cherry Studio、Open WebUI 这类客户端，会带自己的上下文、默认参数和重试行为。

## 推荐实践

- 先用 AI 工作台做“人工验证”
- 确认模型和分组大方向没问题后，再接 API 或客户端
- 复杂问题分开测，不要一次把上下文堆太长
- 真正上线前，还是要回到 API 或客户端路径再验证一遍
