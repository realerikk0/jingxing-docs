# AI 工作台

AI 工作台适用于在不编写代码的情况下，对模型、分组和输出效果进行快速验证。该入口更适合人工测试和效果比较，不替代正式的 API 接入验证。

## 适用场景

- 首次体验井陉
- 比较不同模型的输出风格与响应速度
- 验证某个分组是否适合当前任务
- 为非开发角色提供可直接操作的测试入口

## 与 API 接入的区别

- AI 工作台适合人工验证和对比测试
- API 接入适合 SDK、服务端、脚本和自动化场景

如目标是正式接入业务系统，仍应在工作台验证后，再回到 API 或客户端路径完成最终验证。

## 使用步骤

### 1. 登录控制台

进入控制台后，确认当前账号具备访问工作区的权限。

### 2. 打开 AI 工作台

进入工作台后，建议首先进行最小化验证，不宜在首轮测试中同时使用多个模型和大量复杂参数。

### 3. 选择模型或分组

若当前界面支持切换模型或分组，建议首次仅选择一个已经确认可用的模型进行测试。

### 4. 提交最小测试问题

适合用于首次验证的问题包括：

- 简要介绍产品
- 将一段文字总结为数个要点
- 为一段内容拟定标题

首次测试的目的应集中在以下方面：

- 模型是否正常响应
- 响应速度是否可接受
- 输出风格是否符合预期

## 对比测试建议

### 模型对比

比较不同模型时，建议保持提示词一致，每次仅调整一个变量，以便准确观察速度、稳定性和输出质量差异。

### 分组对比

若界面支持分组切换，可用于判断：

- 分组的稳定性差异
- 分组对不同任务的适配性
- 分组在速度与成本之间的取舍

## 常见误区

### 工作台可用即视为 API 一定可用

工作台与 API 共享同一平台，但接入验证仍应分别完成。工作台可用并不自动代表客户端、SDK 或服务端接入已完全正常。

### 工作台响应慢即视为模型能力不足

响应速度还可能受到当前分组、上下文长度和任务复杂度影响，不应仅以单次测试结果作结论。

### 工作台效果可直接等同于第三方客户端效果

第三方客户端通常会附带自身上下文管理、默认参数和重试机制，因此其实际表现可能与工作台测试结果存在差异。

## 使用建议

- 先用 AI 工作台进行人工验证
- 确认模型与返回效果符合预期后，再进行 API 或客户端接入
- 复杂问题建议拆分测试，避免在首轮测试中引入过多变量
- 正式上线前，应在目标接入路径中再次完成验证
